## Scope decision — 14.5d-1 PARKED (hidden dependency)

EARS 14.5d-1 (cure-item Use-Item UI) requires classify-by-data on `cure_status`, but
`cure_status` lives ONLY on the game-core `ItemDef` content struct (content.rs:170) and is
deliberately absent from the public `item_row` table (schema.rs:118–136) — the
`use_battle_item` reducer's own doc comment (battle.rs:779) states this. The client has no
data path to classify cure items. Mirroring the bait pattern requires an additive
`cure_status` column on `item_row` + content seeding + bindings regen — server-module +
module_bindings changes, outside this slice's declared touches. A hardcoded client item list
would violate ADR-0047 classify-by-data. **Parked** for the supervisor to re-serialize as a
follow-up slice (server additive column first, then a small client slice mirroring the bait
selector verbatim). This slice ships 14.5d-2/3/4 as the coherent mergeable increment.

## Design decisions

- **A. StoreWeather shape** (`client/src/net/store.ts`):
  `interface StoreWeather { readonly tag: string; readonly turnsRemaining: number }` —
  mirrors `StoreStatusEffect`; `turnsRemaining` non-optional because every `WeatherEffect`
  variant carries a u8. `StoreBattle` gains `readonly weather: StoreWeather | null`.
  Tag stays `string` in the store (SDK-agnostic; narrowing happens at the VM/view layer).
- **B. SdkBattleRow structural extension** (`client/src/net/rowConvert.ts`):
  `state.weather?: { readonly tag: string; readonly value: number } | null` (optional →
  existing test factories keep compiling; connection.ts casts structurally, so real rows
  carrying weather flow through). `battleRowToStore` maps to `StoreBattle.weather`
  (absent/undefined/null → null).
- **C. Weather banner is pure-model-first**: `weatherBanner(tag)` pure fn in
  `battleModel.ts` (tag → non-empty label, ''/warn on unknown — same contract as
  `statusBadge`); `BattleViewModel.weather: { label, turnsRemaining } | null`;
  `battleView.ts` gains `#weatherBannerEl` + `#renderWeatherBanner` (thin DOM only,
  `data-testid="weather-banner"`), parallel to the status badge. Does NOT touch
  `#renderActions` (preserves the 13.5e-1 bait-select restore).
- **D. VM-compare guard**: pure `battleVMsEqual(a, b)` in `battleModel.ts` — explicit
  field-by-field compare (bigint via `===`; NEVER JSON.stringify — bigint throws), covering
  outcome, turnNumber, both cards (incl. `status`), skills, bench, baitOptions (incl.
  `count`), canFlee/canSwap/canRecruit, and `weather` (tag + turnsRemaining). `main.ts`
  `refreshBattle()` keeps module-level `lastBattleVM` and skips
  `battleView.refresh(vm)` **only when `battleView.visible`** and equal — guarding on
  visibility (not just cache-reset-on-hide) makes the skip safe against ALL hide paths
  (incl. the Escape bare-hide at main.ts:489 that bypasses refreshBattle's hide branch).
  Reset `lastBattleVM = null` on the refreshBattle hide branch as well.
- **E. Parity guards derive variants from the generated bindings at runtime**
  (probe-confirmed): `StatusEffect.algebraicType.value.variants[].name` yields
  `['Poison','Burn','Paralysis','Sleep','Freeze']` at runtime; same for `WeatherEffect`
  and `BattleOutcome`. The statusBadge/weatherBanner parity tests iterate these — a
  bindings regen with a new variant fails the test with zero hand-list maintenance.
  Test files import bindings read-only (allowed; only src store/rowConvert stay
  SDK-agnostic).
- **F. Outcome literal union**: `BattleOutcomeTag = 'Ongoing' | 'SideAWins' | 'SideBWins'
  | 'Fled'` hand-written in `battleModel.ts`; `BattleViewModel.outcome: BattleOutcomeTag`;
  `buildBattleViewModel` narrows the store's `string` via a small parse (parse-don't-
  validate at the VM boundary; unknown tag → treated per current default behavior).
  `battleView.ts#renderOutcome` switches exhaustively with a `never` check so a union
  growth is a compile error under `tsc --noEmit` (src files ARE typechecked; tests are
  not — that's why the compile gate lives in src). Runtime parity test asserts the
  binding's variant list ⊆ union accepted values.
- **G. ADR-0101**: doc-only, one page — records A–F, the bigint/JSON.stringify rejection,
  the visible-guard rationale, and the 14.5d-1 park with its unblocking path.

## Anti-patterns (named)

hardcoded variant lists without a bindings-derived parity check · JSON.stringify VM compare
(bigint TypeError) · skipping refresh while the view is hidden (stale-hidden trap — hence
the visible-guard) · weather banner churn from missing weather/status in the compare ·
editing generated bindings · runtime module_bindings imports in store/rowConvert ·
`new RegExp(...)` anywhere · touching #renderActions' savedBait restore.

## Risks

- store.test.ts fixtures constructing StoreBattle need `weather: null` — mechanical.
- Existing battleModel tests asserting `outcome` as plain string keep compiling (literals
  narrow); any invalid-outcome test uses StoreBattle input (string) not VM.
- baitOptions.count in the compare is intentional: inventory changes MUST re-render.

## Review refinements (reviewer approve-with-changes + red-team, plan phase)

Binding on tester + implementer:

   `{ tag: state.weather.tag, turnsRemaining: state.weather.value }` — the explicit
   `value → turnsRemaining` rename, parallel to `status.value → turnsRemaining` at
   rowConvert.ts:211. A mapping that drops `.value` (turnsRemaining always undefined)
   must be killed by tests.
2. **(reviewer B-2)** `battleVMsEqual` array compares (skills/bench/baitOptions): length
   check FIRST, then per-element field `===`. Test case: length-differs → not equal.
3. **(red-team 1, proof-of-teeth)** Parity tests anchor before iterating: assert
   `variants.length === 5` (StatusEffect) / `4` (WeatherEffect) / `4` (BattleOutcome)

   never pass vacuously.
4. **(red-team 2+5+6)** Extract the guard decision as a pure
   `shouldSkipBattleRefresh(visible: boolean, lastVm: BattleViewModel | null,
   vm: BattleViewModel | null): boolean` in battleModel.ts — returns true ONLY when
   `visible && vm !== null && lastVm !== null && battleVMsEqual(lastVm, vm)`.
   main.ts uses it; null on either side → never skip. Tests: (nonNull,null)/(null,
   nonNull)/(null,null) never skip · hidden never skips · noWeather↔withWeather VMs
