## Problem (verified in code @ d0c265e)
A false "RLS by `owner_identity`" privacy claim appears in three places; the real posture
(ADR-0040/0046) is: the `inventory` table is **public / world-readable** — there is **no
transport RLS** (`client_visibility_filter` has zero real macro usages in the module).
Owner-scoping is only a **client subscription filter**; per-owner transport RLS is tracked
for **M16**. The danger is a future contributor trusting the comment and shipping a
higher-stakes column into a table they believe is owner-scoped.

- `server-module/src/lib.rs:275-276` — "PUBLIC so a client sees its OWN items (RLS by
  `owner_identity`)" — FALSE.
- `evals/inventory-privacy.eval.mjs:4-6` (header) — "players see their own items via the
  client binding" — misleading (implies owner-exclusive visibility).
- `game-core/src/combat/redteam_m8d_tests.rs:454-458` — "is public with RLS by
  owner_identity, so a player sees only their OWN counts … keep the owner-only RLS filter"
  — FALSE **and self-contradictory** with FINDING 11 (lines 440-451), which correctly says
  the public table leaks counts to all clients.

Plus two ADR fixes:
- `docs/adr/0044-encounter-table-private.md:62` — `species_id: u16` → **u32** (matches
  `EncounterEntryRow` at lib.rs:152 + the species registry width; verified `pub species_id: u32`).
- `docs/adr/0046-player-inventory-model.md` — tighten the visibility prose so it states the
  real posture explicitly, and reconcile the single-stack clause to the parity eval
  (`inventory-single-stack.eval.mjs`, ADR-0054) instead of "convention-only" (ADR-0054
  follow-up (d) delegates this to 7d).

Plus supervisor-directed ADR-index reconciliation:
- `docs/adr/README.md` — add the ADR-0054 row (dev-reducer-release-gating, M8.7b),
  bump "Next free number" 0054 → **0055**, and extend the implementation-ADR range to 0054.
