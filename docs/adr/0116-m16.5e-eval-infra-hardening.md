# ADR-0116 — Eval-infra hardening: append-only snapshot direction, extraction anti-hijack, additive-content coupling

**Status:** Accepted
**Date:** 2026-07-15
**Slice:** m16.5e
**Supersedes:** —
**Amends:** ADR-0076 (spacetime-type snapshot gate), ADR-0103 (BSATN gap finding), ADR-0108 (trade escrow-guard evals)
**Subsystems:** ci-gates
**Decision:** Snapshot gate gains a git-history append-only directional check; escrow-guard extraction sorts, drops `*_tests.rs`, strips strings; every Option content column must have a field-assignment in content.rs's re-seed path.

---

## Context

Ninth-review residuals (spec M16.5 §16.5e) identified three eval-infrastructure gaps:

1. **The spacetime-type snapshot gate (ADR-0076) is exact-match only.** It compares parsed source against the committed baseline — so a *bad re-baseline* (a commit that edits both source and `evals/baselines/spacetime-types.json` together, inserting an enum variant mid-list, reordering variants, or removing a type) sails through. BSATN tags are positional: any non-tail change is a silent wire break for persisted rows (ADR-0103).
2. **`readAllRustSources` in trade-escrow-guards is hijackable.** It concatenates `*.rs` files in platform-dependent (unsorted) order, includes `*_tests.rs`, and `extractFunctionBody` anchors on the first textual `fn <name>(` — which today can be a **string literal**: `economy_tests.rs` contains `"pub fn sell("` inside an assert message. The gate is currently correct only by file-order luck.
3. **No coupling between additive content columns and the re-seed path.** A future `Option<…>` column added to a content table (e.g. via a `..existing`-spread refactor of the row literal) could silently miss its `sync_content_inner` write, leaving stale/None values in re-seeded rows. Today's Option content columns: `species_row.ability`, `item_row.train_stat`, `item_row.cure_status`, `heal_location_row.cost_item_id`.

## Decisions

- **D1 — Enforcement locus (resolves spec D-16.5-3).** All three checks live inside the three existing `*.eval.mjs` files, which already run per-PR via `just eval` inside `just ci`. No justfile edit. Per-PR is the default (a cheap static check that catches the wire break before merge), matching spec decision D-16.5-3.
- **D2 — Fail-open-loud when git/prev-baseline is unavailable.** The directional check SKIPs with a loud `detail` (never silently) when no previous baseline can be resolved. Bounded blind spot: the unchanged exact-match gate still fails any source-vs-baseline drift, and the `ci` job's eval step always checks out `fetch-depth: 0` (the shallow `e2e` job runs only ci-gate-wiring), so the skip can only occur off-CI. Fail-closed would red every offline local `just eval` run for a check whose value is per-PR.
- **D3 — Previous-baseline source: merge-base, with a self-identical fallback to the last transition.** Primary: `git show $(git merge-base HEAD origin/master):evals/baselines/spacetime-types.json` (merge-base, not the origin/master tip, avoids false positives when the branch is behind master). Fallback: origin/master tip, then skip (D2). **If the resolved prev deep-equals the working baseline** (master-push CI: merge-base==HEAD, clean tree), the comparison is vacuous — instead validate the last transition, `HEAD~1:` vs `HEAD:` — closing the direct-push/self-compare hole. Git access uses `execFileSync('git', [constant-args])` (Semgrep-safe; precedent `scripts/okf-export.mjs`).
- **D4 — Coupling scope and shape.** The 16.5e-3 check scans **all of content.rs** (not just the `sync_content_inner` fn body — `heal_location_row` is seeded by `seed_heal_locations_from` outside it). Synced tables are discovered mechanically from `ctx.db.<table>()…update(/insert(/delete(` call sites (multi-line-chain- and index-accessor-tolerant literal regex). The spec's "update-branch write" is realized as **field-assignment presence (`<field>:`) inside the `<StructName> {` row-literal blocks** of content.rs: the upsert tables build one row literal feeding both branches, and the clear-and-reinsert tables (`type_relation_row`, `shop_item_row`, `fusion`) have *no* update branch by design — a bare `update(` requirement would false-positive future Option columns there, while field-assignment catches the actual defect (`..existing`-spread omission) in both shapes. Vacuity guard (zero Option fields discovered → loud FAIL) + four known-instance anchors defend against parser rot (m16.5a lesson).
- **D5 — In-place edits at prefix positions are flagged.** `checkAppendOnly` compares `[name, type]` pairs positionally over the prev-length prefix, so a rename or type change of an *existing* field/variant is flagged even when counts match — an in-place edit of a persisted position is a wire break, not an addition.

## Consequences

- A wire-breaking re-baseline is now caught in PR CI; legitimate evolution is constrained to tail-append (matching additive schema, ADR-0006).
- Guard-site extraction is deterministic (sorted), ignores test sources, and cannot be anchored inside a string literal. Documented residuals (accepted, pre-existing class): raw strings `r#"…"#` are not stripped; a `*/` inside a normal string can corrupt comment-then-string strip ordering.
- A future `Option<…>` content column missing its re-seed write is flagged at PR time. Residual: masking would require a duplicate row-literal of the same struct name — struct names are unique, so the window is same-struct duplication only.
- `docs/adr/README.md` index row for 0116 is deliberately NOT added here (supervisor owns the index; doc-aggregation rule) — reconciled at merge, as with 0114/0115.
- ADR digest regenerated via `just adr-digest` (DIGEST.md + design-corpus.json).
