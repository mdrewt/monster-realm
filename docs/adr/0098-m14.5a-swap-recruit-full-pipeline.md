# ADR-0098 — M14.5a: swap/recruit paths run the full post-turn status/weather pipeline

**Status:** Accepted
**Date:** 2026-07-12
**Slice:** m14.5a
**Supersedes:** —
**Amends:** ADR-0092
**Subsystems:** battle
**Decision:** swap_active and attempt_recruit failure paths run the full run_post_turn_phases pipeline; load_skills() replaces skill_defs_from_rows as the SSOT for skill loading.


**Status:** Accepted  
**Date:** 2026-07-12  
**Slice:** m14.5a (M14.5 eighth-review residuals, closes R1/R3/RT-W14-DESYNC-01)

## Context

Phase B (M14) shipped the full status/weather pipeline (`resolve_full_turn`) only for
`submit_attack`. The swap and recruit-failure reducer paths called raw helpers that skip
every post-turn phase:

- `swap_active` called `resolve_player_swap` → `resolve_enemy_turn` (no DoT, no weather
  chip, no status/weather tick, no StatusApplied write-back).
- `attempt_recruit` (failure path) called `resolve_recruit_failure` with
  `skill_defs_from_rows` (which hardcodes `sets_weather: None` and `applies_status: None`),
  bypassing the entire post-turn pipeline. Because `resolve_recruit_failure` advances
  `turn_number`, status/weather clocks froze while the turn counter advanced — a recruit
  spam loop could defer all DoT/weather damage indefinitely.

The eighth-review (2026-07-11) classified this gap HIGH and named it RT-W14-DESYNC-01.

## Decisions

### D1 — Wire post-turn phases 3–5 into both swap and recruit paths

`resolve_player_swap` and `resolve_recruit_failure` in `game-core/src/combat/resolve.rs`
gain `status: &mut BattleStatusStore` and `sv: &StatusVariance` parameters and run the
same post-turn pipeline as `resolve_full_turn` phases 3–5:

- Phase 3: `apply_post_turn_effects` (DoT)
- Phase 3.5: `apply_weather_damage` (Sandstorm/Hail chip)
- Phase 4: `tick_status` (Sleep/Freeze expire)
- Phase 4.5: write `StatusApplied` events from the enemy retaliation into the store
- Phase 5: `tick_weather` (decrement/expire)

Pre-turn phase 1 (action-block checks) is NOT added to either path. The swap path
intentionally does not block the player — see D3. The recruit path has no player action.

Phase 1.5 (sync `BattleMonster.status` from store) runs before the enemy retaliation
for correctness in test fixtures; in server flow the two are already in sync.

### D2 — `attempt_recruit` uses `load_skills()` not `skill_defs_from_rows`

`skill_defs_from_rows` hardcodes `sets_weather: None` and `applies_status: None`,
creating a game-core / server-module desync (RT-W14-DESYNC-01). The recruit-failure path
now uses `game_core::load_skills()` (the same content-cache path as `submit_attack` and
`swap_active`). `skill_defs_from_rows` remains in marshal.rs for non-battle uses (schema
validation, marshal tests) but is no longer called on any battle-resolution path.

### D3 — Swap is always allowed regardless of player status (D-14.5-1 default b)

ADR-0092 §D3 described a "blocked side's choice converted to Pass" rule that included
`TurnChoice::Swap`. In practice, `swap_active` never consulted `check_action_block` —
Sleep/Paralysis/Freeze were always escapable by switching. This ADR amends D3:

**Swapping is always permitted regardless of the active monster's status condition.**
A swap is an out-of-turn action (it does not consume the player's attack), so the
action-block logic (which gates attacks and passes) does not apply to it. This is the
Pokémon-conventional behavior and the cheaper implementation choice. A pinning test in
`resolve.rs` documents the invariant so any accidental enforcement becomes visible.

The "swap converted to Pass" branch in the previous D3 description is dead in production
and is hereby de-scoped: if it is ever re-evaluated (e.g. for a specific status mechanic
that should trap the player), it will require a new ADR.

### D4 — `swap_active` and `attempt_recruit` build/persist `BattleStatusStore` like `submit_attack`

Both reducers now mirror the submit_attack pattern exactly:
1. Build `BattleStatusStore` from `battle.state` team `.status` fields before calling
   the resolver.
2. Pass `&mut status` to the resolver (which runs phases 3–5 into it).
3. After the resolver, if the battle is still `Ongoing`, write `status.side_a/b` back
   into `BattleMonster.status` fields and persist.
4. If terminal, call `write_back_battle_results` (same as before; status write is skipped
   because the battle is GC'd immediately).

## Consequences

- `resolve_player_swap` and `resolve_recruit_failure` now take 2 extra parameters
  (`&mut BattleStatusStore`, `&StatusVariance`). All callers (5 test call sites in
  game-core, 2 server-module reducers) updated.
- RT-W14-DESYNC-01 gating test updated from "prove the desync" to "pin the fix"
  (the desync assertion is replaced by a positive assertion that `load_skills()` correctly
  sets weather and that the same skill set now produces weather via the recruit path).
- ADR-0092 §D3 body amended in-place (this ADR overrides the swap-conversion clause).
- `skill_defs_from_rows` no longer used on any battle-resolution path; its comment is
  updated. It is NOT removed (used in marshal tests and schema validation).
