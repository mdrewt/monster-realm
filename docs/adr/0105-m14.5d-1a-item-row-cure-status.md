# ADR-0105 — M14.5d-1a: additive `cure_status` column on `item_row`

**Status:** Accepted
**Date:** 2026-07-13
**Slice:** m14.5d-1a (server half of re-serialized 14.5d-1 pair)
**Supersedes:** —
**Amends:** ADR-0101 (closes the "14.5d-1 PARKED" gap documented in its Consequences §)
**Subsystems:** schema-persistence, content, battle
**Decision:** Additive `cure_status: Option<StatusKind>` column on public `item_row`; seeded from content SSOT; `StatusKind` derives `SpacetimeType` (cfg-gated). Unblocks client classify-by-data for cure items (m14.5d-1b).

## Context

The `use_battle_item` reducer (m14e, ADR-0096) validates cure items by reading
`cure_status` from game-core `ItemDef` content — the content SSOT. The reducer
works correctly. However, the public `item_row` SpacetimeDB table did NOT carry
`cure_status`, leaving the client with no data-driven way to build a cure-item
selector list for the battle UI.

ADR-0101 §Consequences documented this as a spec gap (PARKED: "14.5d-1 PARKED
(hidden dependency = SPEC GAP)") with the unblocking path: a small server slice
adds the additive column, then a client slice (m14.5d-1b) mirrors the
bait-selector pattern to build the Use-Item UI.

`StatusKind` (the payload-free status discriminant defined in
`game-core/src/combat/ability.rs`) had no `SpacetimeType` derive, so it could
not be stored in a SpacetimeDB table column. This was the root technical blocker.

## Decision

### D1 — `SpacetimeType` derive on `StatusKind` (cfg-gated, ADR-0003)

`StatusKind` gains `#[cfg_attr(feature = "spacetimedb", derive(spacetimedb::SpacetimeType))]`
immediately above `pub enum StatusKind`, matching the `StatKind` pattern in
`game-core/src/monster/types.rs`. The feature flag ensures the derive is only active
when the `spacetimedb` feature is enabled (by `server-module`), never by `client-wasm`
(feature-isolation gate, ADR-0003).

Variant order (Poison/Burn/Paralysis/Sleep/Freeze) is unchanged — variant order is the
BSATN wire tag and must never be reordered after first publish.

### D2 — Additive column on `item_row` (ADR-0006)

`ItemRow` gains `pub cure_status: Option<StatusKind>` as its LAST field (additive
discipline — never re-order existing fields). Placed after `sell_price`. Seeded 1:1
from `ItemDef.cure_status` in `sync_content_inner` (`server-module/src/content.rs`).

For items without a `cure_status` in the content RON the column holds `None`.
Currently only Antidote (id=3, `cure_status: Some(Poison)`) is a cure item.

### D3 — Reducer unchanged

The `use_battle_item` reducer (ADR-0096) continues to read `cure_status` from the
game-core content cache (`cached_items()`), NOT from the `item_row` DB row. This is
intentional: the reducer must be authoritative against the content SSOT, not against
the table which could theoretically lag a sync. The `item_row.cure_status` column is
strictly client-facing — it allows client subscriptions to classify cure items by data
without an additional game-core content call.

### D4 — `StatusKind` baseline entries

Two eval baselines updated (additive-only, no existing entries changed):
- `evals/baselines/spacetime-types.json`: `StatusKind` enum entry (Poison/Burn/Paralysis/Sleep/Freeze)
- `evals/baselines/table-schemas.json`: `item_row` gains `"cure_status": "Option<StatusKind>"`

These are required by the `spacetime-type-snapshot` and `battle-schema-snapshot` eval
gates respectively.

## Consequences

- The client can now subscribe to `item_row` and filter for `cureStatus !== null` to
  populate a cure-item list in the battle overlay (enables m14.5d-1b).
- `StatusKind` is now a SpacetimeDB type (cfg-gated). It joins `StatusEffect`,
  `StatKind`, etc. in the BSATN type registry. Wire format: enum discriminant, tag
  order = declaration order (Poison=0, Burn=1, Paralysis=2, Sleep=3, Freeze=4).
- Adding a new `StatusEffect` variant still requires updating `StatusKind` (existing
  OCP gate via exhaustive `match` in `StatusKind::matches`); the table wire format
  also requires appending the new `StatusKind` variant last (ADR-0006).
- `docs/knowledge/tables/item_row.md` updated (OKF bundle regenerated).
- Module bindings regenerated: `item_row_table.ts` gains `cureStatus` getter;
  `types.ts` gains `StatusKind` enum export.
- Follow-up: m14.5d-1b (client Use-Item UI) can now be built using the classify-by-data
  pattern, mirroring the bait-selector in the recruit flow.
