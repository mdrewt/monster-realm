# ADR-0117 — Trade SSOT polish: typed authorize rules, symmetric escrow checks, privacy-doc correction, offer TTL reaper

**Status:** Accepted
**Date:** 2026-07-15
**Slice:** m16.5f
**Supersedes:** —
**Amends:** ADR-0106 (trading spine — M-2 residual privacy justification, D4 escrow symmetry), ADR-0108 (trade-reducer-security eval criteria)
**Subsystems:** economy-quests, security-authz, schema-persistence
**Decision:** Respond/confirm checks move to pure `authorize_respond`/`authorize_confirm`; two dead `TradeError` variants deleted; propose escrow checks made symmetric; privacy doc corrected; stale offers reaped by a one-shot TTL reaper with disarm.

---

## Context

The ninth review (M16.5 spec §16.5f) found four trade-subsystem residuals: (1) six `TradeError` variants constructed nowhere while the server shell re-implements them as inline strings already diverged from the variants' Display text, plus a false "state-machine transition functions" claim in `trading/mod.rs`; (2) `propose_trade`'s sufficiency checks are asymmetric — the initiator item loop subtracts `escrowed_item_qty`, the counterparty loop does not, and neither currency check subtracts `escrowed_currency_amount`; (3) the `trade_offer` schema doc falsely cites `player_wallet` as a world-readable precedent (it is PRIVATE must-never-leak, ADR-0015/ADR-0081) and the public `initiator_currency`/`counterparty_currency` fields leak a lower bound on the private balance; (4) `created_at_ms` is written once and never read — no reaper exists, so an idle offer wedges both players' single trade slots until manual cancel or disconnect (DECISION D-16.5-2, default (a) = reaper).

## Decisions

### D1 — Pure `authorize_respond` / `authorize_confirm` in `trading/rules.rs` (16.5f-1)

```rust
pub fn authorize_respond(status: &TradeStatus, is_counterparty: bool) -> Result<(), TradeError>
pub fn authorize_confirm(status: &TradeStatus, is_initiator: bool) -> Result<(), TradeError>
```

- **Role check FIRST, status check second** — preserves the current shell ordering (a non-party caller sees the role error even when the status is also wrong). This ordering is pinned by a unit test (kills a status-first mutant).
- `&TradeStatus` (borrowed): `TradeStatus` is `Clone` but not `Copy`, and the shell uses `offer` after the call — a by-value signature would force a call-site `.clone()`.
- Constructs `NotCounterparty`/`NotPending` (respond) and `NotInitiator`/`NotConfirmedByCounterparty` (confirm).
- The shell calls them **directly in the reducer bodies** (no intermediate helper — the source-scan gates check the `respond_trade`/`confirm_trade` bodies) with `?` propagation and the exact boolean shapes `offer.counterparty == me` / `offer.initiator == me`, `.to_string()`-ing the error via the established `validate_proposal` pattern. A dropped `Result` (`let _ =`) or a wrong-field boolean (`offer.initiator == me` passed as `is_counterparty`) is an authorization bypass; both are pinned by `ea_authorize_respond_01`/`ea_authorize_confirm_01` (trading_tests.rs) and the evolved eval criteria (D5).

### D2 — Delete `MonsterNotOwned` and `InsufficientCurrency { available }` (16.5f-1)

The spec forbids leaving dead variants ("never both sources of truth"). Four of the six dead variants become live via D1. The remaining two are **deleted**, not wired:

- `InsufficientCurrency { available }` — **privacy trap**: the only natural counterparty-side construction site is `propose_trade`'s counterparty balance check, where the error string is returned to the *initiator*; embedding `available` there would leak the counterparty's PRIVATE wallet balance (ADR-0015/ADR-0081 must-never-leak). Deleting the variant prevents any future caller from reintroducing the leak. Propose-time currency errors stay inline and balance-free. If a future auction-house slice adds a pure-layer wallet check, introduce a new amount-free variant instead.
- `MonsterNotOwned` — the shell's ownership rejection (`build_cards`) carries the offending monster id in its message; a bare variant loses that information, and no authorize path performs ownership checks (they need DB context). Deleted as intentional dead-code removal.

The `trade_error_display_is_meaningful` test's `cases` vec is trimmed in lockstep (both rows), keeping full Display branch coverage of the remaining variants.

### D3 — Symmetric propose escrow subtraction (16.5f-2, amends ADR-0106 D4 commentary)

`propose_trade` subtracts escrow on **both sides for both asset kinds**: the counterparty item loop gains the same `escrowed_item_qty` subtraction as the initiator loop, and both currency checks subtract `escrowed_currency_amount`. Each escrow sum chains **both** btree indexes (`initiator().filter(owner)` + `counterparty().filter(owner)`) so an offer where the owner sits on either side is counted — the same chain the initiator item loop already uses. Under ADR-0106 D4 (one active offer per player, already enforced above via `validate_proposal`), every one of these subtractions is provably 0 at this point in the reducer; they are kept and commented as such for the auction-house extension where a player may hold multiple concurrent offers.

### D4 — Per-offer one-shot TTL reaper with disarm (16.5f-4, DECISION D-16.5-2 option (a))

- **Pure rule (SSOT, game-core rules.rs):** `pub const TRADE_OFFER_TTL_MS: i64 = 3_600_000` (1 hour — a tunable liveness constant, not a game-design constraint; long enough for human negotiation, short enough to unstick abandoned escrow) and `pub fn is_offer_stale(created_at_ms: i64, now_ms: i64) -> bool` using `saturating_sub` + `>=` (a future `created_at_ms` yields a negative delta → not stale; extremes cannot panic).
- **Scheduled table** `trade_offer_reaper_schedule` colocated with its reducer in `server-module/src/trading.rs` (ADR-0056 exception, mirroring `pvp_deadline_schedule`/`movement_tick_schedule`). PRIVATE — the annotation prevents client schedule manipulation; the underlying facts (`trade_id`, creation time) are already visible via the public `trade_offer` row, so no information is being hidden. Fields: `scheduled_id` (pk auto_inc), `scheduled_at: ScheduleAt`, `trade_id: u64` with `#[index(btree)]` for disarm. No redundant `created_at_ms` — the reaper re-reads the live offer.
- **Armed once in `propose_trade`** by capturing the insert return (auto_inc `trade_id` — the ADR-0072 capture-insert pattern) and scheduling `ScheduleAt::Time` at `created_at_ms*1000 + TRADE_OFFER_TTL_MS*1000` micros. Computing the deadline from the ms-floored `created_at_ms` (not raw micros) guarantees `now_ms - created_at_ms >= TTL` at fire time — no ms-truncation edge where a fired one-shot no-ops and the offer lives forever.
- **Reaper** `trade_offer_reaper`: scheduler-only guard (`ctx.sender != ctx.identity()`), re-reads the offer (gone → `Ok(())` no-op), re-checks `is_offer_stale(offer.created_at_ms, now_ms(ctx))` before deleting. Reaps **regardless of status** — Pending and ConfirmedByCounterparty both hold escrow and wedge both players' slots; deletion is the identical terminal-GC path `cancel_trade` uses (no assets move, ADR-0106 D3). **No TTL refresh on accept** — the initiator is the party who must act after acceptance, and refreshing would let a counterparty keep a dead offer alive.
- **Disarm on early termination:** `disarm_trade_reaper(ctx, trade_id)` deletes the pending schedule row(s) and is called at all four offer-deletion sites (`respond_trade` reject, `cancel_trade`, `confirm_trade`, `cancel_trades_on_disconnect`). Without this, a propose→cancel griefing loop accumulates one orphaned schedule row per cycle for the full TTL hour (unbounded scheduler backpressure). This deliberately **extends** the pvp precedent (pvp does not disarm) because pvp deadline rows live ≤60 s while trade rows would live 1 h — a 60× amplification. One-shot rows that do fire are auto-deleted by SpacetimeDB — a documented platform guarantee, not an assumption: "Scheduled reducers delete the row after execution" (SpacetimeDB docs, schedule-tables §Row Lifecycle; the row is visible while the reducer runs). The reaper therefore must NOT self-delete (it would race the runtime's post-execution delete); ADR-0109 D7 is the in-repo precedent.
- **Mutation-gate note:** the reaper body needs a `ReducerContext` and is not unit-testable in-crate (same known gap as `pvp_deadline_reaper`); its invariants are pinned by source-scan tests/eval criteria instead. Nightly `cargo-mutants` may report live mutants in the reaper body — accepted, matching the pvp precedent.

### D5 — Eval criteria evolve from token-presence to delegation-shape (amends ADR-0108)

`evals/trade-reducer-security.eval.mjs` pinned literal tokens inside the shell bodies (`TradeStatus::Pending`, `offer.counterparty != me`, …). After D1 those tokens legitimately move to game-core, so the criteria evolve in lockstep with the shell change (same commit — the gate must never be red on a commit): `RESPOND_ROLE`+`RESPOND_STATUS` → `RESPOND_AUTHORIZE` and `CONFIRM_ROLE`+`CONFIRM_STATUS` → `CONFIRM_AUTHORIZE`, each requiring the direct call **plus `?` propagation plus the correct `offer.<field>` in the argument window** (call-presence alone is gate-transparent to the dropped-Result and wrong-field attacks — red-team findings 1/2). A new `AUTHORIZE_RULES` criterion keeps the status tokens pinned at their new home in `rules.rs`. New reaper criteria cover: schedule armed in `propose_trade` after the offer insert (position check), scheduler-only guard, `is_offer_stale` re-check, offer deletion, and ≥4 `disarm_trade_reaper` call sites. Every changed/new checker ships with bad/good proof-of-teeth fixtures that bite.

### D6 — Privacy-doc correction (16.5f-3, amends ADR-0106 M-2)

The `trade_offer` schema doc and ADR-0106's M-2 residual falsely cited `player_wallet` as an existing world-readable precedent. Corrected: `player_wallet` is PRIVATE must-never-leak; `inventory` is genuinely world-readable pending transport RLS (that half of the claim was accurate). The public `initiator_currency`/`counterparty_currency` fields leak a **lower bound** on the offering party's private balance to all subscribers — recorded as an **accepted bounded exposure** (offered amounts only, never the full balance; propose-time error strings never echo balances per D2). A related pre-existing (M15a-era) vector is also recorded as part of this accepted exposure: `propose_trade`'s distinct counterparty-insufficiency error strings allow an initiator to binary-probe a counterparty's balance/item count (≤64 calls for an exact balance); fixing it means degrading propose-time UX to a generic error and is deferred to the transport-RLS/privacy milestone. `trade_offer` is flagged for the same transport-RLS treatment as `inventory`/`player_wallet` when per-row RLS lands.

## Consequences

- Respond/confirm authorization now has exactly one source of truth (game-core), unit-tested without a `ReducerContext`; shell error strings are the typed variants' Display text (previously silently diverged).
- `TradeError` loses two variants — an API-shape change to a `pub` enum with zero production constructors (verified by grep); the Display test shrinks accordingly.
- New table + reducer ⇒ regenerated client bindings, knowledge bundle, and appended (append-only) `table-schemas.json`/`spacetime-types.json` baselines.
- Abandoned offers now clear after 1 h, releasing both players' trade slots without manual cancel/disconnect.
