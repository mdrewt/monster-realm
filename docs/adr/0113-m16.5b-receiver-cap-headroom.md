# ADR-0113 — Receiver-cap headroom check in confirm_trade (reject, don't destroy)

**Status:** Accepted
**Date:** 2026-07-15
**Slice:** m16.5b
**Supersedes:** —
**Amends:** ADR-0108 (M15c trade conservation) — closes the receiver-side cap clamping gap
**Subsystems:** economy-quests, schema-persistence, ci-gates

**Decision:** `confirm_trade` calls new pure `check_headroom` (game-core/trading/rules.rs) before any mutation. Item credit exceeding `MAX_ITEM_STACK` or currency credit exceeding `MAX_BALANCE` → `Err`, full rollback. `MAX_ITEM_STACK` moved to game-core.

---

## Context

M15c (ADR-0108) established the trade conservation invariants: every asset debit has a matching credit (`consume_one`/`grant_item`, `spend_currency`/`grant_currency`). This closed the duplication and burning paths.

However, the credit-side functions are **infallible clamps**:
- `grant_item` silently caps the receiver's stack at `MAX_ITEM_STACK = 9999` (monotone-cap invariant from ADR-0059). A receiver holding 9,980 items who receives 50 ends up with 9,999, not 10,030. The 31 excess items vanish with no `Err` and no log.
- `grant_currency` silently saturates at `MAX_BALANCE = 999_999_999`. A receiver at cap who receives any currency sees no change.

The sender's side is correctly guarded: `consume_one` returns `Err` on empty stock; `spend_currency` returns `Err` on insufficient balance. But there was no symmetric guard on the receiver side.

The ninth review (M16.5 residual) identified this as a conservation failure: the debit path is fallible and the credit path is not, so a full-stack receiver trading for items destroys value without signal. The reject-not-clamp invariant (ADR-0003 / server-authoritative discipline) requires `Err`, not silent clamping, when an operation cannot complete fully.

`MAX_ITEM_STACK` was previously a `pub(crate)` constant inside `server-module/src/inventory.rs`. Because `check_headroom` lives in `game-core` (pure rule layer, no I/O), the constant must live there too.

## Decision

### D1 — `check_headroom` in game-core

A new pure function `check_headroom` is added to `game-core/src/trading/rules.rs`. It takes eight parameters:
- `initiator_receives_items: &[TradeItem]` — items moving TO the initiator (= counterparty's offer)
- `initiator_current_stacks: &[ItemStack]` — initiator's current inventory snapshot for those item IDs
- `initiator_receives_currency: u64` — currency moving TO the initiator
- `initiator_balance: u64` — initiator's current wallet balance
- `counterparty_receives_items: &[TradeItem]` — items moving TO the counterparty
- `counterparty_current_stacks: &[ItemStack]` — counterparty's current inventory snapshot
- `counterparty_receives_currency: u64` — currency moving TO the counterparty
- `counterparty_balance: u64` — counterparty's current wallet balance

The function checks each receiver independently. For items: `current.saturating_add(qty) > MAX_ITEM_STACK` → `Err(TradeError::ItemStackCapExceeded { item_id })`. For currency: `balance.saturating_add(credit) > MAX_BALANCE` → `Err(TradeError::CurrencyCapExceeded)`. Returns `Ok(())` if all headroom checks pass.

`#[allow(clippy::too_many_arguments)]` is applied — 8 arguments is unavoidable for the symmetric two-sided × four-component structure; grouping into a struct would add boilerplate for a single call site.

### D2 — `MAX_ITEM_STACK` moved to game-core (ADR-0113 SSOT)

`MAX_ITEM_STACK: u32 = 9999` is now `pub const` in `game-core/src/trading/rules.rs`. `server-module/src/inventory.rs` imports it via `use game_core::MAX_ITEM_STACK;` and removes its local definition. This makes the constant a domain invariant in the pure rule layer — `check_headroom` can reference it without a crate boundary crossing, and any future pure-layer logic has a single authoritative source.

### D3 — `ItemStack` struct in game-core

A new `pub struct ItemStack { pub item_id: u32, pub current_count: u32 }` is added alongside `check_headroom`. It represents one item-stack snapshot passed into the headroom check. The server-module caller assembles the slices by querying `ctx.db.inventory()` immediately before calling `check_headroom`.

### D4 — Two new `TradeError` variants

`game-core/src/trading/types.rs` adds:
- `ItemStackCapExceeded { item_id: u32 }` — "receiver item stack for item {item_id} would exceed the maximum stack size"
- `CurrencyCapExceeded` — "receiver currency balance would exceed the maximum balance"

Both are wired into the existing `Display` impl and the `trade_error_display_is_meaningful` test.

### D5 — Placement in `confirm_trade`

The `check_headroom` call is inserted in `server-module/src/trading.rs → confirm_trade`, immediately after the battle-interlock re-assertion (D1 of ADR-0112) and BEFORE `build_swap_plan`. This ensures:
1. No mutations occur before the headroom check (the check is a pre-flight, not a rollback).
2. The TOCTOU window is handled: if the receiver fills their inventory between `respond_trade` and `confirm_trade`, the call fails cleanly.

The call converts `TradeError` to `String` via `.to_string()` and calls the existing `log_reject` helper, matching the pattern of all other guards in `confirm_trade`.

### D6 — Proof-of-teeth

`server-module/src/trading_tests.rs` adds `ea_conservation_headroom_01_confirm_trade_calls_check_headroom`: a source-scan test that extracts the text of `confirm_trade` (using `strip_rust_comments_trading`) and asserts it contains a `check_headroom(` call.

`evals/trade-conservation.eval.mjs` extends to a 7th criterion: `HEADROOM_CHECK` — bad fixture (no call → must flag) + good fixture (with call → must pass) + production source check.

`game-core/src/trading/rules.rs` adds 9 unit tests covering: initiator item at cap, initiator currency at cap, counterparty item at cap, counterparty currency at cap, exactly-at-cap item (stack=MAX_ITEM_STACK qty=1 → Err), exact headroom item (9980+19=9999 → Ok), exact currency headroom, empty inputs, and new-receiver (current=0 qty=MAX_ITEM_STACK → Ok).

## Considered alternatives

**A — Make `grant_item` and `grant_currency` fallible (return `Result`).**
Rejected: these functions are called from many reducers (recruit, shop buy/sell, quest reward, battle XP). Making them fallible would require propagating `?` everywhere and rewriting every call site. The cap-overflow scenario is specific to trading (the only reducer that credits a receiver who has no influence over their own stock level); other callers can guarantee the count stays in range by construction.

**B — Cap-with-remainder (transfer what fits, reject only the excess).**
Rejected: partial asset transfers violate the atomic swap model. The sender would lose M items while the receiver gains only the fitting subset. This is indistinguishable from asset destruction from the sender's perspective and violates the conservation invariant (ADR-0108).

**C — Integrate the check into `build_swap_plan` rather than as a separate pre-flight.**
Rejected: `build_swap_plan` is a pure function that takes ownership snapshots as input, not live DB state. Feeding receiver inventory into it would widen its signature and break all callers. The pre-flight placement (before `build_swap_plan`) achieves the same ordering guarantee with no signature change.

## Consequences

- **No silent value destruction**: a trade where ANY receiver cannot fully absorb their incoming assets is rejected in full, before any mutation. The sender's assets are untouched.
- **SSOT for MAX_ITEM_STACK**: the constant is now in game-core; any future pure-layer logic that needs the stack cap imports it from there.
- **Symmetric fallibility**: the credit path (`check_headroom → Err`) is now as strict as the debit path (`consume_one`, `spend_currency`), closing the conservation gap.
- **Marginal breaking scenario**: two players with near-full inventories/wallets who previously received a silent partial fill now get an explicit rejection and must reduce their inventories to complete the trade. This is the correct behavior.
