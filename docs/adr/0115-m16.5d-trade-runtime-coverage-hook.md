# ADR-0115 — Trade runtime coverage: test-hook dispatch + e2e round-trip + escrow-guard tail

**Status:** Accepted
**Date:** 2026-07-15
**Slice:** m16.5d
**Supersedes:** —
**Amends:** ADR-0107 (M15b trade client UI), ADR-0108 (M15c trade evals tail) — closes trade-reducer test coverage gap
**Subsystems:** client-ui, ci-gates

**Decision:** Add `window.__mrTrade` write-side test hook; two-context Playwright e2e for propose→confirm trade flow with monster conservation check; TR-13 guard site for `attempt_recruit` + RT-SEC-02b string-literal stripping in `bodyHasGuard`.

---

## Context

M15b (ADR-0107) shipped the trade client UI with KeyU overlay + `TradeView` DOM shell. M15c (ADR-0108) completed trade evals, adding 11 guard-site checks in `trade-escrow-guards.eval.mjs` (reject_if_monster_in_trade × 7 reducers, escrowed_item_qty × 2, escrowed_currency_amount × 2). However, the full propose→respond→confirm round-trip had no automated runtime coverage:

1. **No reducer dispatch hook.** The `window.__game()` test-hook pattern (established M5a) exposes read-only state snapshots for e2e validation. The trade subsystem needs write-side dispatch — proposing an offer, a second player responding, and a first player confirming — to exercise the full lifecycle.

2. **TR-13 missing from escrow guards.** The `attempt_recruit` reducer in `server-module/src/taming.rs` calls `escrowed_item_qty` (bait-escrow pre-flight check, added in M16.5a/ADR-0112). However, the guard site was absent from the `GUARD_SITES` table in `trade-escrow-guards.eval.mjs` and was incorrectly tagged TR-12, colliding with `use_battle_item`'s TR-12. The eval counted 11 sites; CI could not audit `attempt_recruit`'s bait-escrow check independently.

3. **False-positive trap in guard-site scanning.** The `trade-escrow-guards.eval.mjs` static analyzer counts guard call sites by grepping `reject_if_monster_in_trade` and `escrowed_item_qty` in source code. A Rust string literal like `log::info!("reject_if_monster_in_trade(x)")` would be counted as a real guard call (RT-SEC-02b — the "string literal false-positive" trap). No such literal currently exists in the codebase, but the eval should strip string/comment contents before counting.

## Decision

### D1 — `window.__mrTrade` test-hook dispatcher

A new object on `window` exposes the four trade reducers as async wrapper functions (async for Promise-based await in Playwright), plus two subscription query methods:

```typescript
interface MrTrade {
  proposeTrade(initiator_id: string, counterparty_identity: string, initiator_offer: TradeItemVec, counterparty_offer: TradeItemVec): Promise<void>;
  respondTrade(offer_id: string, response: 'Accept' | 'Reject'): Promise<void>;
  confirmTrade(offer_id: string): Promise<void>;
  cancelTrade(offer_id: string): Promise<void>;
  allTradeOffers(): Array<{offer_id: string; initiator: string; counterparty: string; status: string}>;
  allPlayers(): Array<{identity: string; character_name: string}>;
}
```

All BigInt fields (identities, offer_ids, monster_ids) are serialized as strings for the `postMessage` boundary with Playwright workers. The hook mirrors the `window.__game` pattern: included in all builds (no DEV gate), consistent with existing precedent. A JSDoc comment warns that the `MrTrade` interface in `client/e2e/trade-full.spec.ts` must be kept in sync manually (the wasm boundary does not auto-export TS types).

### D2 — Two-context Playwright e2e flow (`trade-full.spec.ts`)

A new test file in `client/e2e/trade-full.spec.ts` (Playwright @serial) launches two distinct `chromium.launch()` instances, each with its own SpacetimeDB identity. Two test cases:

**m16.5d-1:** Hook existence. Verifies `window.__mrTrade` is present and exports all six methods.

**m16.5d-2/3/4:** Full propose→respond→confirm flow. Player A connects (granted a Flameling at level 5 via starter). Player B connects. Player A calls `proposeTrade(A_monster_id, B_identity, [Flameling], [])`. Player B calls `respondTrade(offer_id, 'Accept')`. Player A calls `confirmTrade(offer_id)`. Assertion: before = after = 2 total monsters (A and B each own 1; no monsters lost or duplicated).

Atomic confirm re-reads live rows (ADR-0106 D4) so no stale-data exploit is possible; the conservation assertion gates this at runtime.

### D3 — TR-13 guard site addition + RT-SEC-02b false-positive hardening

**D3a — `attempt_recruit` escrow guard registered as TR-13.** The `attempt_recruit` reducer in `server-module/src/taming.rs` already calls `escrowed_item_qty` (shipped in M16.5a as part of the battle↔trade interlock, ADR-0112). However, it was absent from the `GUARD_SITES` table in `trade-escrow-guards.eval.mjs` and was previously tagged TR-12, which collided with `use_battle_item`'s TR-12. This eval added `['attempt_recruit', 'escrowed_item_qty', 1, 'TR-13']` as the 12th guard site, resolving the tag collision and making the guard auditable by CI. Guard-site count: 11 → 12 (TR-13).

Two proof-of-teeth fixtures added to `trade-escrow-guards.eval.mjs`:
- Good fixture: `attempt_recruit` body WITH `escrowed_item_qty(` → `bodyHasGuard` returns ok:true.
- Bad fixture: `attempt_recruit` body with only `consume_one(`, no `escrowed_item_qty(` → `bodyHasGuard` returns ok:false (kills a regression where the bait-escrow pre-flight check is removed).

**D3b — `bodyHasGuard` string-literal stripping (RT-SEC-02b).** The `bodyHasGuard(body, guardName)` function in the eval now strips Rust string and comment contents before searching for the guard call. Pattern: `//.*` (line comments) and `"[^"]*"` (string literals) are replaced with empty space. This prevents false positives from log messages or documentation strings containing guard names. The evaluation remains sound: a guard call is always outside string/comment delimiters.

One additional tooth added: `RT-SEC-02b` verifies the stripping logic by including a known string literal in a test file, confirming it is NOT counted as a guard site.

## Consequences

- Full trade round-trip (propose→respond→confirm) is now gated by an e2e test exercising both reducer dispatch and server state mutations.
- Guard-site count in `trade-escrow-guards.eval.mjs` increases 11 → 12 (TR-2..TR-13).
- `bodyHasGuard` is hardened against false positives from string literals and comments containing guard names.
- Monster conservation is asserted at the final confirm step, validating atomic re-read semantics (ADR-0106 D4).
- The `window.__mrTrade` pattern establishes a reusable write-side test-hook convention for future e2e closures.
