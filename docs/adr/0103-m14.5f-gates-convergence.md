## Context and problem statement

Two gates tasks were identified as residuals at M14 Phase B close (ADR-0097):

**14.5f-1 — BSATN schema-compat proof**: M14b (ADR-0093) and M14d (ADR-0095) added optional
`BattleMonster.status` and `BattleState.weather` fields, annotated `#[serde(default)]` to
handle RON/serde round-trips with old rows. The question was whether this annotation is
sufficient to cover the SpacetimeDB persistence path (BSATN codec). The plan for option (a)
— a live smoke run using a pre-M14b server module — was infeasible: `spacetime 2.6.0 CLI`
has no `--features` passthrough, so building the module without the `dev_reducers` feature
gates out the battle-related reducers needed to create test fixtures.

**14.5f-2 — Convergence net widening**: The existing `netcode-convergence` eval (ADR-0013,
M8.8d) exercised SeqCanonical vs Arrival over 16 fixed seeds. No coverage for randomized
intent mixes (Jump, all four directions, queue-cap bursts), warp-crossing scenarios under a
lossy link, or battle-locked characters mid-stream.

## Considered alternatives

**14.5f-1:**
- Option A (live pre-M14b smoke): Build the server module before the status/weather fields
  were added, publish it, run a reducer, check deserialization. Ruled out: spacetime 2.6.0
  CLI `spacetime build` has no `--features` flag; `dev_reducers` is required to create
  `battle_state` rows in test reducers.
- Option B (static analysis + finding documentation): JS eval using `readFileSync` + string
  search to verify `#[serde(default)]` is co-located on each target field, `SpacetimeType`
  derive is present on each target struct, the RON/serde default tests exist in
  `m14b_tests.rs`, and the codec-gap finding is documented in the eval name+detail.
  **Chosen.**

**14.5f-2:**
- Extend the existing scenario to 128 seeds: does not exercise different intent mixes.
- Add `random_scenario()` generator + `warp_scenario_under_link()` + `battle_locked` world
  model: wider surface with meaningful novel inputs. **Chosen.**

## Decision outcome
- Chosen: Option B for 14.5f-1; comprehensive scenario expansion for 14.5f-2.
- Consequences:
  - **FINDING (14.5f-1):** `#[serde(default)]` covers the serde/RON codec path only.
    `BattleMonster` and `BattleState` persist via `SpacetimeType` / BSATN, which is a
    position-based codec (all fields required in order; no missing-field defaults). Additive
    schema compatibility for existing rows is handled at the **SpacetimeDB engine level**
    (publish without `--delete-data`), not by the BSATN codec. The serde annotation is
    correct and necessary for content-loading paths but is not the mechanism that makes
    BSATN-persisted rows backward-compatible.
  - `bsatn-compat-smoke.eval.mjs` documents this finding and verifies co-location of the
    serde annotation + SpacetimeType derive on both structs + RON test existence.
  - `ServerWorld` gains `battle_locked: bool` per character, mirroring `movement.rs:207-220`.
    The queue is left intact when locked (freeze = drain suppressed, not enqueue blocked).
  - `random_scenario()` uses `tick_seed` exclusively (no OS entropy, per ADR-0055).
  - Warp non-vacuity: `warp_convergence` is set false if no seed produces a reorder, guarding
    against jitter=0 regressions.
  - `convergencePasses` in `netcode-convergence.eval.mjs` now checks 7 criteria; 3 new
    proof-of-teeth (E/F/G) ensure no criterion can be silently removed.
  - ADR next-free: **0104**.
