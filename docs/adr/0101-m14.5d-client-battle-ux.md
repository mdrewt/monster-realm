# ADR-0101 — M14.5d: Client battle UX completeness (weather banner, parity guards, VM-compare)

**Status:** Accepted  
**Date:** 2026-07-13  
**Slice:** m14.5d (M14.5 eighth-review residuals §14.5d; ships EARS 14.5d-2/3/4)

## Context

M14.5 spec §14.5d required four EARS: cure-item Use-Item UI (14.5d-1), weather
banner threading (14.5d-2), bindings-derived parity guards (14.5d-3), and
VM-compare refresh guard (14.5d-4). The slice touches `client/src/**` only
(excluding `module_bindings/**`).

**14.5d-1 PARKED — undeclared server dependency (spec gap).** `cure_status`
lives only on the `game-core` `ItemDef` content struct (content.rs:170) and is
deliberately absent from the public `item_row` table (schema.rs:118–136); the
`use_battle_item` reducer doc comment (battle.rs:779) states this explicitly.
A classify-by-data client UI (per ADR-0047) has no data path without an additive
`cure_status` column on `item_row` plus content seeding plus bindings regen —
all server-module changes outside this slice's declared touches. The spec's
"client/src/** only" touches declaration is therefore a spec gap. Unblocking
path: a server slice adds the additive column (mirroring `bait_type` on
`item_row`), then a client slice mirrors the bait-selector pattern verbatim.

This ADR records the three shipped decisions (D1–D4) and the documented gaps.

## Decisions

### D1 — Weather pipeline: store → VM → view

`StoreWeather { readonly tag: string; readonly turnsRemaining: number }` is
added to `client/src/net/store.ts`, mirroring the existing `StoreStatusEffect`
interface. `StoreBattle` gains `readonly weather: StoreWeather | null`. Tag
stays `string` in the store layer — the store is SDK-agnostic; narrowing is
deferred to the VM boundary.

`SdkBattleRow` in `rowConvert.ts` is extended structurally with
`state.weather?: { readonly tag: string; readonly value: number } | null`
(optional so existing test factories keep compiling). `battleRowToStore` maps
`state.weather` → `StoreWeather` with an **explicit rename**:
`turnsRemaining: state.weather.value` — parallel to the
`status.value → turnsRemaining` mapping at rowConvert.ts:211. The object is
tested with `value === 0` (zero-falsy trap: `?.value || null` would misfire here;
truthiness is checked via `!= null`, never `?.value || null`).

`battleModel.ts` adds `weatherBanner(tag: string): string` — a pure function
returning a non-empty label for each known `WeatherEffect` variant, `''` plus
`console.warn` on unknown (same contract as `statusBadge`).
`BattleViewModel.weather: { label: string; turnsRemaining: number } | null`.
`buildBattleViewModel` derives it from `StoreBattle.weather`.

`battleView.ts` gains `#weatherBannerEl` and `#renderWeatherBanner` (thin DOM,
`data-testid="weather-banner"`, textContent-only for XSS safety), parallel to
the status badge. The render path is cleared on `null` / hide. `#renderActions`
is not touched, preserving the 13.5e-1 bait-select restore.

### D2 — BattleOutcomeTag literal union and exhaustive switch

`BattleOutcomeTag = 'Ongoing' | 'SideAWins' | 'SideBWins' | 'Fled'` is
hand-written in `battleModel.ts`. `BattleViewModel.outcome: BattleOutcomeTag`.

`buildBattleViewModel` narrows `StoreBattle.outcome` (a `string`) at the VM
boundary. An unknown outcome tag → `console.warn` + return `null` (same null-VM
hide path as corrupt-team). `decideBattleOverlay` is not changed — it continues
to operate on `StoreBattle.outcome: string`.

`battleView.ts #renderOutcome` switches exhaustively over `BattleOutcomeTag`
with a `never`-check default arm. The pre-existing
`default: text = \`Battle ended: ...\`` interpolation arm is **replaced** (not
supplemented) — the never arm is genuinely unreachable at runtime because
unknown outcomes null out in `buildBattleViewModel` upstream.

New server variant ⇒ runtime parity test red ⇒ developer grows the union ⇒
`tsc --noEmit` forces every switch site. This is the compile-gate pattern; test
files are not typechecked, so the gate lives in src.

### D3 — Bindings-derived parity guards (proof-of-teeth)

`statusBadge`, `weatherBanner`, and `BattleOutcome` parity tests derive the
variant list at runtime from the generated enum shape:
`X.algebraicType.value.variants[].name`. Each test anchors before iterating:

| Enum | Expected length | Known-member probe |
|------|----------------|--------------------|
| `StatusEffect` | 5 | `'Poison'` |
| `WeatherEffect` | 4 | `'Rain'` |
| `BattleOutcome` | 4 | `'Ongoing'` |

Length anchor + known-member check together ensure an empty or truncated array
can never pass vacuously. Test files import generated bindings read-only;
`store.ts` and `rowConvert.ts` do not import from `module_bindings` (SDK-agnostic
invariant preserved).

### D4 — VM-compare refresh guard

`battleVMsEqual(a: BattleViewModel, b: BattleViewModel): boolean` is a pure
function in `battleModel.ts`. Comparison is explicit field-by-field:
`bigint` fields via `===`; arrays length-first then per-element field `===`
(`skills`, `bench`, `baitOptions` incl. `count`); `weather` null-tolerant
(`== null`); `status` tag+turnsRemaining. **`JSON.stringify` is rejected** —
it throws `TypeError` on `bigint` (`battleId` is `bigint`).

`shouldSkipBattleRefresh(visible: boolean, lastVm: BattleViewModel | null,
vm: BattleViewModel | null): boolean` is a pure predicate (also in
`battleModel.ts`): returns `true` only when `visible && vm !== null &&
lastVm !== null && battleVMsEqual(lastVm, vm)`. Null on either VM side → never
skip.

`main.ts refreshBattle()` keeps a module-level `lastBattleVM: BattleViewModel |
null` and calls `shouldSkipBattleRefresh(battleView.visible, lastBattleVM, vm)`
**before** calling `battleView.refresh(vm)`. The visibility check is the primary
defense: while the view is hidden the guard never skips, so all hide paths are
safe — including the Escape bare-hide (main.ts:489) that bypasses refreshBattle's
hide branch entirely.

Overlay lifecycle state (`dismissedBattleId`, `battleSynced`) is updated **before**
the guard so skips cannot desync the state machine. `lastBattleVM` is reset to
`null` in the hide branch, the Escape handler, and `resetPredictionState`.
Reconnect is safe: `resetPredictionState` runs synchronously before the
microtask-deferred `flushBatch`.

## Consequences

- `StoreBattle` gains `weather: StoreWeather | null`; all existing test factories
  constructing `StoreBattle` add `weather: null` (mechanical).
- `BattleViewModel.outcome` changes from `string` to `BattleOutcomeTag`; literals
  narrow cleanly, but any test passing an invalid outcome string via `StoreBattle`
  (not VM) continues to compile.
- `baitOptions[i].count` is included in the VM compare intentionally — inventory
  changes must re-render.
- **Gap: Sleep `turnsRemaining` is not in `BattleMonsterCardVM`** (badge-only).
  A future sleep-countdown display must add `turnsRemaining` to the card VM AND
  to `battleVMsEqual`; omitting the compare addition would cause the refresh guard
  to suppress tick re-renders (ADR-0100-D6-style gap note).
- Two unknown weather tags with equal `turnsRemaining` compare as equal via `''`
  labels — zero visual impact (no banner renders for either), and the parity test
  prevents this for all known variants.
- `console.warn` on unknown outcome or weather tag fires per-batch (~20 Hz) until
  bindings are regenerated — intentional loudness.
- ADR index and CHANGELOG are updated by the supervisor at merge, not this slice.
