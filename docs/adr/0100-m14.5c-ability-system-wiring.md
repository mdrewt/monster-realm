# ADR-0100 — M14.5c: Wire passive-ability system end-to-end

**Status:** Accepted  
**Date:** 2026-07-12  
**Slice:** m14.5c (M14.5 eighth-review residuals §3; closes D-14.5-2 default a)

## Context

ADR-0094 (M14c) introduced the passive ability rule layer in `game-core`:
`AbilityStore`, `apply_entry_ability`, `apply_ability_modifiers`, and two
`AbilityEffect` variants (`StatusImmunity` / `EntryHeal`). The implementation
was intentionally **not wired** into the server or the resolve functions at that
time — the module doc comment recorded this deferral explicitly.

The eighth-review spec (M14.5 §14.5c) closes the deferral: the ability column
must exist on `species_row`, species content must assign abilities exercising
all known effect kinds, and every battle reducer path must construct and thread
an `AbilityStore` through the resolve functions.

## Decisions

### D1 — Additive `ability: Option<u32>` column on `SpeciesRow`

A new `ability: Option<u32>` field is appended as the **last** field on
`SpeciesRow` (additive column, ADR-0006). `None` means "no passive ability".
The `game_core::Species` struct already carries `#[serde(default)]
pub ability: Option<u32>`; the server-side `SpeciesRow` now mirrors it.

No migration is needed for existing rows: SpacetimeDB treats the absent column
as `None` on reads before the first `sync_content` run post-deploy.

### D2 — Species content: two ability assignments covering all effect kinds

`game-core/content/species/000-core.ron`:

| Species | id | ability_id | Effect |
|---------|----|------------|--------|
| Flameling | 1 | 1 | `StatusImmunity { immune_to: Burn }` (Flame Body) |
| Sproutlet | 3 | 3 | `EntryHeal { denom: 4 }` (Regeneration) |
| Tidalin | 2 | — | none |

This covers both shipped `AbilityEffect` variants, pinning them with content-driven
end-to-end tests (EARS 14.5c-3).

### D3 — `build_ability_store` pure helper in `marshal.rs`

A new pure function `build_ability_store(side_a_ability_ids, side_b_ability_ids,
ability_defs) -> AbilityStore` resolves `Option<u32>` ability IDs against the
loaded `AbilityDef` registry. Unknown IDs resolve silently to `None` (a stale def
does not crash battle start). The function lives in `marshal.rs` — consistent with
other pure DB-type-to-game-core-type translations.

### D4 — `AbilityStore` threaded through all three resolve functions

`resolve_full_turn`, `resolve_player_swap`, and `resolve_recruit_failure` each
gain an `abilities: &AbilityStore` parameter as their final argument. Inside:

- `resolve_full_turn`: Phase 0 (pre-existing per ADR-0094) calls
  `apply_ability_modifiers(state, status, abilities)` before Phase 1 action-blocking.
- `resolve_player_swap`: calls `apply_entry_ability(state, swap_side, abilities, status)`
  after `set_active` + Switch event, before Phase 1.5 (post-turn phases).
- `resolve_recruit_failure`: calls `apply_ability_modifiers` after the advance-turn
  check, before Phase 1.5.

### D5 — Server reducers build `AbilityStore` from DB species rows

All five battle-path reducers now construct an `AbilityStore` before calling
the resolve functions:

| Reducer | Builds AbilityStore? | Calls apply_entry_ability? |
|---------|---------------------|--------------------------|
| `start_battle` | yes (during team build) | yes, for side A+B active=0 |
| `begin_encounter` | yes (during team build) | yes, for side A+B active=0 |
| `submit_attack` | yes (species lookup per slot) | no (per-turn only) |
| `swap_active` | yes (species lookup per slot) | via resolve_player_swap |
| `attempt_recruit` | yes (species lookup per slot) | via resolve_recruit_failure |

`start_battle` and `begin_encounter` additionally call `apply_entry_ability` for
the initial active on both sides before inserting the `Battle` row, using a
temporary `BattleStatusStore` reconstructed from the team and written back.
Status write-back mirrors the per-turn pattern in `submit_attack`.

### D6 — Auto-switch on KO is a follow-up gap (documented)

`apply_entry_ability` is NOT called on auto-switch (monster KO'd mid-turn). The
gap means `EntryHeal` does not fire when a monster is force-switched in after a
KO. `StatusImmunity` still takes effect the next turn via `apply_ability_modifiers`.
This gap is acceptable for this slice and is tracked for Phase C.

### D7 — `ability.rs` doc comment amended

The doc comment that read "intentionally NOT wired into resolve_full_turn in this
slice" is superseded by D4. The comment is removed; the wiring is now complete.

## Consequences

- `SpeciesRow` gains one nullable column — no write-incompatibility for existing
  rows; additive per ADR-0006.
- Client bindings regenerated (`just gen`); `species_row_table.ts` now includes
  `ability: __t.option(__t.u32())`.
- Eval baseline (`evals/baselines/table-schemas.json`) updated for `species_row`.
- All callers of the three resolve functions (resolve.rs test module, m14a/m14b/
  m14d/m14e/m14_5b/redteam test files, external test files) updated to pass an
  empty `AbilityStore` as the last argument.
- `species_from_row` in `marshal.rs` now threads `row.ability` through instead
  of hardcoding `None`.
