# ADR-0099 — M14.5b: StatusApplied event carries emission-time slot; drop write if not conscious

**Status:** Accepted  
**Date:** 2026-07-12  
**Slice:** m14.5b (M14.5 eighth-review residuals §2, closes RT-M14.5B-01)

## Context

Phase 4.5 of `run_post_turn_phases` writes `StatusApplied` events from the combat action
into `BattleStatusStore`. At the time of `StatusApplied` emission (Phase 2, inside
`resolve_one_attack`), the event carries `{ side, status }` but NOT the targeted slot.
Phase 4.5 must then infer the correct slot.

The m14.5a review pass (ADR-0098) fixed the acute form of this bug — it captured
`active_slot_a/b = state.side_X.active` at the **start** of `run_post_turn_phases`,
before phases 3/3.5 could trigger an auto-switch. That approach is correct for the
slot-selection invariant, but leaves two gaps:

1. **No `slot` in the event.** The correct slot is inferred indirectly from early
   capture timing rather than encoded in the event itself. This makes the causal
   relationship implicit and fragile to future restructuring.

2. **No consciousness guard.** If the targeted monster survives the attack (so
   `StatusApplied` is emitted) but then **faints from DoT or Sandstorm/Hail chip in
   Phase 3/3.5**, the current code still writes the status to the (now-fainted)
   monster's slot. The spec requires this write be dropped.

The eighth-review spec (M14.5 §2, 14.5b-1) mandates both: the event carries the slot,
and Phase 4.5 drops the write when the targeted monster is no longer conscious.

## Decisions

### D1 — Add `slot: u32` to `BattleEvent::StatusApplied`

`StatusApplied` gains a `slot: u32` field containing the team index of the targeted
monster captured **at the moment of emission** inside `resolve_one_attack`. This makes
the target unambiguous in the event stream, independent of subsequent `state.side_X.active`
changes.

```rust
// Before:
StatusApplied { side: SideId, status: StatusEffect }

// After:
StatusApplied { side: SideId, slot: u32, status: StatusEffect }
```

The `slot` is `state.side_X.active` at the moment `StatusApplied` is pushed in
`resolve_one_attack`, captured BEFORE any faint-or-auto-switch for the DEFENDER side
(the `!fainted` guard at that call site ensures the attacked monster did not faint from
this attack, so the active index has not yet changed for the defender side at emission).

### D2 — Phase 4.5 reads slot from the event; drops write if not conscious

`run_post_turn_phases` no longer captures `active_slot_a/b` at entry. Instead, Phase 4.5
extracts `(side, slot, status)` triples directly from `StatusApplied` events. Before
writing into `BattleStatusStore`, it checks whether the targeted monster is still
conscious:

```rust
let is_conscious = state.side_X.team
    .get(slot as usize)
    .map(|m| !m.is_fainted())
    .unwrap_or(false);
if is_conscious {
    store_vec[slot as usize] = Some(new_status);
}
// else: drop — targeted monster fainted between emission and Phase 4.5
```

Dropping a StatusApplied write when the target fainted is semantically correct: applying
a status to a fainted monster is a no-op in all downstream paths (DoT never runs for
fainted monsters, the monster doesn't participate in future turns). The drop keeps the
store consistent with observable game state.

### D3 — `active_slot_a/b` early-capture removed

The `active_slot_a/b` capture lines added by the m14.5a review pass are removed — the
event-slot approach fully subsumes them. The Phase-4.5 comment is rewritten to explain
D1 and D2 rather than the superseded capture timing.

### D4 — types.rs `StatusApplied` comment corrected

The previous comment stated "Since no auto-switch can fire for the non-fainted target,
`state.side_X.active` is stable between the emit (Phase 2) and the application (Phase
4.5)." This was INCORRECT: the non-fainted targeted monster cannot auto-switch from
THIS attack, but DoT (Phase 3) or weather chip (Phase 3.5) CAN faint the target and
trigger an auto-switch AFTER emission. The comment is replaced with the correct
explanation: the slot is encoded in the event at emission time (D1), and Phase 4.5
reads it directly.

## Consequences

- All code that constructs or matches `BattleEvent::StatusApplied` must include `slot`.
  Seven test files updated; the field is `u32`, serde round-trips cleanly.
- `BattleEvent`s are not persisted to the database (server module discards `_events`);
  no schema migration needed.
- `RT-M14.5A-01` (m14.5a gating test): the assertion that the fainted slot carries
  `Some(Burn)` changes to `None` — consistent with D2's drop rule. The key invariant
  (backup/switch-in slot has no status) is unchanged and still bites.
- New test `m14_5b_tests.rs` pins the `resolve_full_turn` path (the most common path):
  near-lethal Burn hit + Sandstorm chip faint → both the fainted target slot and the
  switch-in slot carry `None` in the store.
