# ADR-0127 — PvP runtime coverage, DEV-gated test hooks, and SDK-boundary enum exhaustiveness

**Status:** Accepted
**Date:** 2026-07-18
**Slice:** m17.5f
**Supersedes:** —
**Amends:** ADR-0115 (reverses D1's ungated-hooks decision)
**Subsystems:** client-ui, ci-gates

**Decision:** DEV-gate `window.__game`, `window.__mrTrade`, `window.__mrPvp` test hooks; add PvP two-context e2e with turn-advance + interlock evals; add SDK-enum-exhaustiveness eval + fail-soft narrowTag boundary check.

---

## Context

M16.5d (ADR-0115) shipped `window.__mrTrade` test-hook dispatch as an ungated fixture ("included in all builds"), establishing the write-side hook pattern for e2e two-player scenarios. M16c (ADR-0121) added PvP evals but no PvP e2e (no runtime coverage above unit math); the spec assigned m17.5f to close that gap (EARS 17.5f-1/17.5f-2) and add an SDK-boundary enum exhaustiveness gate (EARS 17.5f-3). M17.5f also resolves the forward-cleanup note in main.ts (ADR-0115 code review D1, lines 1064–1066) — DEV-gating all three hooks reduces attack surface at zero e2e cost (the dev server is `npm run dev` where `import.meta.env.DEV === true`). Meanwhile, `rowConvert.ts:525` performs a silent `as` cast of `trade_offer.status.tag` into a union type; with no runtime narrowing, a future server-added variant is silently received as the raw string, possibly violating the consumer's exhaustive-switch assumptions (the JSON-protocol boundary is append-only per BSATN stability, but client handling is not enforced).

## Decision

### D1 — DEV-gate all three window test hooks

`window.__game`, `window.__mrTrade`, and `window.__mrPvp` are window assignments inside an `if (import.meta.env.DEV) { ... }` guard block at `main.ts:1211–1214` (reversed from ADR-0115 D1's ungated stance). The guard is implemented via **Vite's define-replacement** of `import.meta.env.DEV` (replaced with `false` in `vite build`), followed by the **minifier's dead-branch constant-fold elimination** — NOT Rollup tree-shaking (the mechanism is critical: a `vite build --minify false` bundle would retain the dead-code assignments, but server-side `ctx.sender` authz still prevents privilege escalation; callers can only act as themselves, verified across all five PvP reducers). Production builds drop the hooks with zero e2e impact: the Playwright webServer is spawned as `npm run dev`, where `import.meta.env.DEV` is `true` at runtime.

Reconciliation direction: ADR-0115 D1's sentence "included in all builds (no DEV gate)" is **superseded** by this decision.

### D2 — SDK-boundary enum exhaustiveness: eval-diff + fail-soft narrowTag

Three mechanisms:

**1. Eval-diff (the load-bearing gate).** New eval `evals/sdk-enum-exhaustiveness.eval.mjs` statically parses `client/src/module_bindings/types.ts` for `__t.enum("EnumName", { ... })` variant lists (handling both unit variants like `Fire: __t.unit()` and payload variants like `Rain: __t.u8()` — only the KEY matters for tag matching) and compares them line-by-line against the `HANDLED_ENUM_VARIANTS` registry exported from `client/src/net/rowConvert.ts` (stripped of comments to avoid false positives). Server-added variant → types.ts regen widens the enum → eval RED until the registry is consciously updated. Scope: boundary enums only (registry keys), not every generated enum (YAGNI). Registry currently covers: `TradeStatus`, `ChallengeStatus`, `BattleOutcome`, `Affinity`, `StatusKind`, `WeatherEffect`, `ActionState`, `Direction` (8 enums; `PvpAction` explicitly excluded — it is a write-direction reducer arg, never read from a row by rowConvert).

**2. Coupling ratchet (type system + eval).** Widening a `HANDLED_ENUM_VARIANTS` const entry widens the `narrowTag<T>` inferred generic at its call site (`:525`, the ONLY narrowTag usage). Assigning a wider union to the narrower `StoreTradeOffer.status` field (hardcoded `'Pending' | 'ConfirmedByCounterparty'` literal union, established in ADR-0114) is a TypeScript compile error — forcing a deliberate store-union + consumer decision before a server variant is shipped. The eval additionally pins that `:525` references `HANDLED_ENUM_VARIANTS.TradeStatus` textually (adjacency coupling, C4), preventing a raw `as` cast with a stray const reference from satisfying the gate.

**3. Runtime narrowTag check (fail-soft boundary).** New function `narrowTag(raw: string, known: readonly T[], enumName: string): T` in `rowConvert.ts` centralizes the silent cast. On unknown tag (future server-added variant), it logs via `console.warn` (one call per unknown value, unbounded warn volume but bounded by table size) and returns the raw value — **NEVER throws** (flushBatch has NO per-listener isolation; a throw would kill the entire subscription batch). This is a fail-soft net, not the primary gate.

**Explicitly REJECTED:** replacing `{ tag: string }` structural interfaces with generated type-only unions. This violates rowConvert's SDK-decoupling header (store never imports SDK; tests build plain objects) and does not catch the defect — structural widening keeps TypeScript green regardless.

### D3 — PvP runtime coverage: two-context e2e flows

New tests validate the full PvP lifecycle without CLI/server instrumentation:

**pvp-full.spec.ts** (m17.5f-1): two separate `chromium.launch()` instances, distinct SpacetimeDB identities. Flow: A → B challenge → B accepts → both subscribe to `battleById(id)` → A + B submit matched actions → `turnNumber === turnNumber0 + 1` strict predicate (NO terminal-escape hatch; a deadline forfeit does NOT advance turnNumber and would fake the test) → 30s budget → immediate B disconnect (forfeit) → A waits terminal → zero-sum rating sql assertion (winner +Δ wins+1, loser −Δ losses+1, sum==2000). Deadline headroom: 30s turn budget, immediate forfeit sequencing keeps ≥25s of PvP_TURN_DEADLINE_MS=60s. Kills: acceptChallenge no-op, submitPvpAction no-op, terminal-escape bypass, wrong-player winloss, non-zero-sum rating, zero-delta path.

**trade-interlock.spec.ts** (m17.5f-2): two contexts, file named to sort AFTER golden/recruit (eliminating presenceCount==2 cross-contamination). Paired positive-control design: (b) B successfully proposes a trade (all guards pass) and it appears; (b-cancel) B cancels (proving G2 active-offer guard is clearable); (c) A enters a wild battle then B repeats the IDENTICAL proposal — if it inserts a row, `reject_if_in_battle` is missing. Sql ABSENCE assertion at the end with a bounded settle window. Kills: missing battle interlock guard, false-green stale-offer confound (positive control failure).

## Consequences

- Production `vite build` drops all three window hooks via minifier dead-branch elimination (attack-surface reduction, zero e2e cost).
- New eval gates client enum-handling drift complementary to server-side `spacetime-type-snapshot` (server append-only BSATN vs client exhaustive handling).
- `narrowTag` centralizes the one cast site (`:525`), auditable and logged; flushBatch-safe fail-soft boundary.
- E2e suite grows 8 → 10 specs (approx. +50s local wall-clock, workers:1).
- Registry entries WITHOUT narrowTag call sites (bare-string store fields) have eval-only awareness — accepted limitation, documented here (prevents unused-const noise).
- ADR next-free: **0128**

## Accepted limitations and residuals

1. **Unbounded console.warn per unknown tag:** each unknown-tag row issues one warn (no dedup); bounded by table size, not warn-once dedup (which would collide with the existing warn-assertion tests).
2. **Registry entries without narrowTag sites:** e.g., `ChallengeStatus` feeds bare-`string` store field; eval-only enforcement via C2/C3. Store-union changes out of scope (ADR-0114 already narrowed TradeStatus).
3. **StatusEffect deliberately excluded:** appears only on reducer args, never on row-read boundaries.
4. **submitPvpAction unknown tag assumption:** BSATN encode/server decode rejects it (SDK behavior, not client-verified).
5. **PvP deadline-forfeit e2e deferred:** 60s+ real wait; disconnect-forfeit path (`pvp-full.spec.ts`) covers the escrow semantics.
6. **`--minify false` caveat:** production intent is `vite build` with default minify; a future CI step building a production bundle without minify would retain the hooks, but server authz still holds.

- Payload-carrying enum (Rain: __t.u8()) → parses correctly, good tooth.

- Known tag passes through typed.
- Unknown tag returns raw AND logs (spy asserts).
