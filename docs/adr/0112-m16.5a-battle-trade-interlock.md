# ADR-0112 — Battle↔trade interlock (both directions) + vacuous-revival gate

**Status:** Accepted
**Date:** 2026-07-14
**Slice:** m16.5a
**Supersedes:** —
**Amends:** ADR-0106 (M15a trade spine) — closes the trade-to-battle interlock gap; ADR-0109 (M16a PvP spine) — extends coverage to PvP `opponent_identity` side
**Subsystems:** trading, battle, ci-gates, security-authz

**Decision:** `propose_trade` and `confirm_trade` now call `reject_if_in_battle` for every offered monster on both sides, chaining both `player_identity()` and `opponent_identity()` btree indexes to cover PvP side-B participants. `m7b_2_owner_change_mid_battle_spec_gap` gains a source-scan body asserting the abort-on-owner-change contract and the new interlock. `evals/spec-gap-revival.eval.mjs` rejects revived anchors whose stripped body contains no `assert`.

---

## Context

M15a (ADR-0106) established the trade escrow model: `start_battle` calls `reject_if_monster_in_trade` so a monster in an active trade offer cannot enter battle. This is one direction of the interlock. The ninth review (2026-07-14) identified that the OTHER direction was missing:

- `propose_trade` and `confirm_trade` in `trading.rs` had **zero battle checks**.
- A monster already in an Ongoing battle could be offered in a trade and, if confirmed, would be transferred out of the battle. Every terminal battle path routes through `write_back_party_hp`, which calls `ctx.db.monster().monster_id().find(mid)` and then checks `owner_identity != battle.player_identity` before writing HP. On an owner change, the function returns `Err`, which rolls back the terminal outcome (win/loss/flee/recruit), leaving the battle permanently Ongoing. `already_in_battle` (in `start_battle`) would then brick the player forever.
- The in-code comment at `write_back_party_hp` calling this path "unreachable" (on the basis of TR-11/ME-1 escrow guards) was FALSE — those guards prevent trading a monster INTO a battle (via `start_battle`), not trading one OUT of an Ongoing battle.

Additionally:
- M16a (ADR-0109) added a PvP battle model where a monster can appear on the `opponent_identity` side (Side B) of a battle row. The pre-existing `reject_if_in_battle` call pattern in `evolution.rs` only chains `player_identity()`, so a Side-B monster was invisible to the guard.
- `m7b_2_owner_change_mid_battle_spec_gap` had been "revived" (un-ignored) by M15a but its body was comment-only — zero assertions — making the revival vacuous. `evals/spec-gap-revival.eval.mjs` only checked that the function existed and was not `#[ignore]`, not that its body actually exercised anything.

## Decision

### D1 — Reject-not-clamp in propose_trade and confirm_trade

Both `propose_trade` and `confirm_trade` now call `reject_if_in_battle` for every monster on both sides (initiator and counterparty) before executing any trade logic. The guard is in `guards.rs` (ADR-0056 SSOT); no new logic is introduced.

Placement:
- `propose_trade`: AFTER ownership/inventory pre-flight checks, BEFORE `build_cards`. Prevents a proposal from being created with a battling monster.
- `confirm_trade`: BEFORE `build_swap_plan`. Defense-in-depth against the TOCTOU window between `respond_trade` (status → ConfirmedByCounterparty) and `confirm_trade` (atomic swap). A new battle may start in this window.

### D2 — PvP side-B coverage via opponent_identity index

Both call sites chain BOTH btree indexes:
```rust
ctx.db.battle().player_identity().filter(owner)
    .chain(ctx.db.battle().opponent_identity().filter(owner))
```

`reject_if_in_battle` already checks both `party_monster_ids` AND `opponent_monster_ids` in each battle row it sees. The only missing piece was that the iterator didn't cover battles where `opponent_identity == owner` (PvP Side B). This is now fixed. The `opponent_identity` btree index was added in ADR-0109; the `chain()` call is O(log n) on both sides.

### D3 — m7b_2 body: source-scan assertions

`write_back_party_hp` cannot be called from game-core (circular dependency: server-module depends on game-core). The test uses `include_str!` to load `server-module/src/battle.rs` and `server-module/src/trading.rs` at compile time, then asserts:

1. The abort-on-owner-change check exists in `write_back_party_hp` (`owner_identity != battle.player_identity` → `return Err`).
2. The abort appears BEFORE any `ctx.db.monster().monster_id().update(` so the other player's monster row is left untouched.
3. `trading.rs` calls `reject_if_in_battle` at least twice (once per reducer).
4. `propose_trade` chains `opponent_identity()` (PvP side-B coverage).

### D4 — Vacuous-revival gate in spec-gap-revival.eval.mjs

A new function `revivedAnchorHasAssert(testSrc)` is exported. It extracts the `m7b_2_owner_change_mid_battle_spec_gap` function body, strips line comments, and returns `true` iff the stripped body contains at least one `assert`. The real-file gate now also calls this check: a revived anchor (not `#[ignore]`) with a comment-only body fails CI.

Proof-of-teeth (V1/V2/V3): vacuous body → `false`; body with `assert!` → `true`.

## Consequences

- The battle↔trade interlock is now bidirectional: `start_battle` guards trade→battle; `propose_trade`/`confirm_trade` guard battle→trade.
- PvP participants on either side (Side A via `player_identity`, Side B via `opponent_identity`) cannot have their monsters traded away during an ongoing PvP battle.
- `m7b_2_owner_change_mid_battle_spec_gap` is genuinely revived with 5 real assertions.
- The vacuous-revival class is closed: future spec-gap closures that leave comment-only bodies will fail CI.
- `reject_if_in_battle` call count in `trading.rs` >= 2, verified by EA-TRADE-BATTLE-04 (mutation kill: guard present in only one reducer → test fails).
- ADR next-free: **0113**
