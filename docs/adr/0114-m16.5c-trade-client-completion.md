# ADR-0114 — Trade client completion: overlay symmetry, typed TradeStatus, render hygiene

**Status:** Accepted
**Date:** 2026-07-15
**Slice:** m16.5c
**Supersedes:** —
**Amends:** ADR-0107 (M15b trade client UI) — closes three residuals from the M16.5 ninth-review
**Subsystems:** client-ui, ci-gates

**Decision:** Close three M16.5 residuals: e2e test for KeyQ/KeyH/KeyG overlay guards; `StoreTradeOffer.status` narrowed to a literal union with exhaustive switch; `TradeView` clears stale feedback on state change and sets `btn.disabled=this.#pending`.

---

## Context

M15b (ADR-0107) shipped the trade client UI. M16.5 (ninth review) identified three residuals:

**16.5c-1 — Overlay symmetry gap.** The M16b review pass (51e6f5d) added `!tradeView?.visible` guards to the KeyB/KeyI/KeyE handlers. KeyQ (shop), KeyH (heal), and KeyG (quest log) were fixed in the same pass but had no e2e coverage confirming the reverse direction (open trade via KeyU, then press Q/H/G, assert only `#trade-overlay` is visible). The guard code was already correct; the missing gate was an e2e test proving it.

**16.5c-2 — `StoreTradeOffer.status` typed as `string`.** `store.ts` declared `status: string` with a JSDoc comment. `tradeModel.ts`'s `deriveActionsAndLabel` used an if/else chain on a `string` parameter. TypeScript had no knowledge of the valid variants, so a future server addition of a third `TradeStatus` variant would silently be handled by the else branch (wrong behavior, no compile error). The fix narrows both the store interface and the function parameter to a literal union; the exhaustive switch then produces a TS control-flow "function lacks ending return statement" error at the switch site for any new variant.

**16.5c-3 — Render hygiene in `TradeView`.** Two bugs: (a) `#renderActions()` clears `actionsEl.innerHTML` then creates fresh buttons — each fresh button defaults to `disabled=false`, so a mid-flight re-render (while `#pending=true`) re-enables the buttons and opens a double-send window. (b) `render()` never touched `#feedbackEl`; a "Trade accepted!" message from a prior status would persist into a new status, and stale messages from a prior trade session would bleed into a new offer.

## Decision

### D1 — e2e test for overlay symmetry (16.5c-1)

A new Playwright test `'trade open: G/Q/H keys do not open overlays (16.5c-1)'` is appended to `client/e2e/trade.spec.ts` inside the existing `describe.serial` block. It opens the trade overlay via `'u'` key, then presses `'g'`, `'q'`, and `'h'` in turn, asserting after each that `#shop-overlay`, `#quest-log-overlay`, and `#heal-overlay` are hidden and `#trade-overlay` is still visible.

The guard code (`!tradeView?.visible`) was already present from M16b; this test provides proof-of-teeth. The e2e test is the authoritative gate because the guard logic lives in the DOM shell (`main.ts` keyboard handler), which is excluded from unit coverage.

### D2 — `TradeStatus` literal union and exhaustive switch (16.5c-2)

`TradeStatus = 'Pending' | 'ConfirmedByCounterparty'` is exported from `tradeModel.ts`. `StoreTradeOffer.status` in `store.ts` is narrowed to this union. The SDK boundary in `rowConvert.ts` casts `row.status.tag as 'Pending' | 'ConfirmedByCounterparty'` (trust-the-server pattern; the server is the authoritative source of variant names). `deriveActionsAndLabel` is rewritten from if/else to two exhaustive switch blocks (`isInitiator` path and counterparty path) with explicit return type annotation; TypeScript's control-flow analysis makes any uncovered variant a "Function lacks ending return statement" compile error.

The `[m15b-TM-6d]` and `[m15b-TM-10a]` tests in `tradeModel.test.ts` previously passed invalid status strings to exercise the unknown-branch; these are updated to use only valid statuses (the invalid-string path is now a TypeScript compile error, not a runtime branch). A new `[m16.5c-TM-12a]` test exhaustively covers all four role×status cells.

### D3 — Render hygiene in `TradeView` (16.5c-3)

Two changes in `tradeView.ts`:

**D3a — Feedback clearing on state change.** A private `#lastRenderKey: string | null` field tracks `"${vm.tradeId}-${vm.statusLabel}"`. In `render()`, when the key changes (or transitions from/to `no-trade`), `#feedbackEl.textContent = ''` is called before updating the DOM. Same-state re-renders leave existing feedback intact (e.g. a "Trade accepted!" from the current status persists until status advances). `hide()` and the `no-trade` branch both reset `#lastRenderKey` to `null` so the next trade session starts clean.

**D3b — `btn.disabled = this.#pending`.** In `#renderActions()`, each freshly-created button is assigned `btn.disabled = this.#pending` immediately after creation. This means a re-render while a reducer Promise is in-flight creates buttons that are already disabled, closing the double-send window.

Three unit tests cover D3 in `tradeView.test.ts` (`@vitest-environment happy-dom`):
- `[m16.5c-TV-1]`: re-render while `#pending=true` → all buttons have `disabled=true`
- `[m16.5c-TV-2]`: feedback persists on same-status re-render; cleared when statusLabel changes
- `[m16.5c-TV-3]`: feedback cleared when new offer arrives after `no-trade` state

## Consequences

- TypeScript compile error on any future third `TradeStatus` variant → forces the developer to handle it.
- Double-send via mid-flight re-render is closed.
- Stale feedback cross-status and cross-session is closed.
- The overlay-symmetry invariant is now gated by an e2e test (not just guard code).
