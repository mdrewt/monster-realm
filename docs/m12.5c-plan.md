# M12.5c Plan ÔÇö Client zone-sync robustness

**Slice:** `feat/m12.5c-zone-sync-robustness`
**ADR reserved:** 0074

## Problem (┬ž 12.5c of the spec)

Four verified bugs in the M11c warp/zone-sync system:

1. **Reconnect-strand (12.5c-1):** Server deletes character on disconnect and recreates at zone 0 on rejoin. Client `onReconnect` only resets prediction; `rawMap` stays at the old zone. Warp detection is edge-triggered (`onUpdate` only), so a character INSERTED at zone 0 after reconnect never fires `onOwnWarp`. Result: own character invisible (render filter skips it), camera at origin, prediction on wrong map ÔÇö permanent until page reload.

2. **Idle-remote strand (12.5c-2):** `store.resetCharacters()` in the warp handler clears ALL characters. The global subscription never re-inserts idle remotes that don't move. They become permanently invisible after a warp.

3. **Mutation-before-parse (12.5c-3):** `renderer.setMap(rawMap)` calls `TileMap.fromRaw()` ÔÇö which can throw ÔÇö AFTER `set_active_zone`, `rawMap` assignment, and `store.resetCharacters()`. Comment claims atomicity but it's false.

4. **rAF loop mortality + reconcile containment (12.5c-4):** A `wasm`/`apply_move` throw inside `frame()` kills the rAF loop because `requestAnimationFrame(frame)` is the LAST statement. `predictor.reconcile()` is outside the listener's try block ÔÇö a throw starves sibling listeners.

## Architecture Decision

**Approach: Purely state-based zone sync.** The `onBatchApplied` reconcile listener checks `own.row.zoneId !== rawMap.zone_id` each batch. When they differ, call `switchZone(newZoneId)`. This single mechanism handles BOTH reconnect (character INSERTED at zone 0) and live warp (character UPDATED to new zone).

Keep `onOwnWarp` (idempotent: `switchZone` no-ops if already at target zone). The `onOwnWarp` fires earlier (in the SDK row callback) for a lower-latency live-warp response; the batch listener catches the reconnect case. No double-switch because `switchZone` checks `newZoneId !== rawMap.zone_id`.

**No `store.resetCharacters()` on zone switch.** `RenderResolver.resolve()` already filters by `currentZoneId`; old-zone characters are invisible at render time. New-zone characters (incl. idle ones) remain in the store from initial subscription. This is the render-filter option (spec 12.5c-2 Option A).

## Concrete changes

### `client/src/main.ts`
1. Add `import { TileMap } from './render/map'` (for pre-validate before mutations).
2. Move `renderer` to module scope: `let renderer: WorldRenderer | undefined`.
3. Move `resetPredictionState` to module scope (it only references module-level vars).
4. Add module-scope `switchZone(newZoneId)`:
   - Guard: `if (newZoneId === rawMap.zone_id) return` (idempotent)
   - `zone_map(newZoneId)` Ôćĺ `TileMap.fromRaw(newRawMap)` (validate first) Ôćĺ `set_active_zone` Ôćĺ `rawMap = newRawMap` Ôćĺ `renderer?.setMap(rawMap)` Ôćĺ `resetPredictionState()`
   - try/catch: log + keep current zone on failure
5. First `onBatchApplied` (reconcile): add zone-sync check + wrap entire body in try-catch.
6. Simplify `onOwnWarp` callback to just `switchZone(newZoneId)`.
7. Frame loop: move `requestAnimationFrame(frame)` to a `finally` block; wrap loop body in `try/catch`.
8. Add `setRawMapZoneForTest: (zoneId) => { rawMap = zone_map(zoneId); }` to `window.__game()` for e2e proof-of-teeth.
