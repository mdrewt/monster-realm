## 1. Resolved design decisions (→ ADR-0126)

**D1 (D-17.5-1): TTL = 120_000 ms (2 min).** A challenge is an interactive prompt to a player online at send time (guard 3), and a Pending row locks both parties out of new challenges (guards 5b/6) — the TTL directly bounds the *lockout* window for AFK/disconnected challengers. Trade's 1 h TTL is for offers that survive sessions; challenges must not. Tunable, `>=` boundary.

**D2 (amended per W3): `CHALLENGE_TTL_MS` + `is_challenge_stale` live in the EXISTING `game-core/src/combat/pvp.rs`.** That module's doc already claims "PvP rules live exactly once in the functional core"; it is verified UNPINNED (zero include_str!/eval reads of combat/pvp.rs, combat/mod.rs, game-core lib.rs). A new root `game_core::pvp` module (v1 plan) was REJECTED: it would split the PvP SSOT and collide with the existing module name. ranking.rs (scan-pinned by m17.5d) and trading/rules.rs (wrong domain) also rejected.

**D3 (reworded per F5): 17.5e-2 cooldown DEFERRED to M19; record, don't build.** A cooldown needs a second private table (state must outlive the deleted challenge row) → more schema/bindings surface and its own GC question. HONEST SCOPE: the TTL bounds only the AFK/disconnected-attacker lockout; an ACTIVE attacker bot can re-challenge the instant its row is reaped/declined — TTL does not bound that; recourse is the M19 decline-cooldown/mute primitive. Record this residual verbatim in ADR-0126. Smallest coherent mergeable increment = reaper alone.

**D4: disarm at ALL FOUR deletion sites** (accept_challenge, decline_challenge, cancel_challenge, cancel_challenges_on_disconnect). Reaper no-ops on missing rows and challenge_id is auto_inc (no ABA), so the disconnect-site disarm is hygiene — but "every deletion site disarms" keeps EA-CHR-02 un-gameable, mirrors EA-REAPER-02, and avoids orphaned scheduler rows. F8 verified the four sites are EXACTLY the deletion set (whole-tree grep; no `.update(` sites → no status ever leaves Pending; no re-arm path).

**D5: NO status recheck in the reaper.** Non-Pending rows never persist (ADR-0109 D6; F8 re-verified). A `status != Pending` branch is dead code and an unkillable-mutant magnet. Defenses: existence check + `is_challenge_stale` recheck. Trade-reaper parity.

**D6 (inherited):** deadline from ms-floored `created_at_ms`×1000 + TTL×1000 (ADR-0117 D4); NO self-disarm (races runtime post-execution delete, ADR-0109 D7); schedule table PRIVATE; scheduler-only guard `ctx.sender != ctx.identity()`. STRUCTURAL INVARIANT (F4): the schedule table must never grow a timestamp field — the field set {scheduled_id, scheduled_at, challenge_id} is what guarantees staleness cannot be computed from client-suppliable args (row's `created_at_ms` only).

**D7 (new, F6): early-fire no-op = accepted leak residual.** A genuinely early fire (backwards wall-clock step) hits the `!is_challenge_stale` branch → `Ok(())` → the runtime consumes the one-shot row → the Pending row leaks until manually declined/cancelled (both delete+disarm). Accepted for template fidelity (trade reaper identical); ADR-0117 D4 ms-flooring eliminates the known systematic cause. No re-arm branch (pathological-clock refire loop risk + YAGNI). Record in ADR-0126 residuals.
