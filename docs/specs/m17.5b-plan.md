**Slice:** m17.5b · **ADR:** 0123 (supervisor-assigned; Phase E re-verifies next-free before writing) ·
**Branch:** feat/m17.5b-trade-conservation · **Plan status:** REVIEWED (reviewer + red-team lenses closed; all findings adopted — see §Findings log)
**EARS:** 17.5b-1 (debits-before-credits), 17.5b-2 (netted currency headroom), 17.5b-3 (executed conservation proof-of-teeth)

## Root cause (verified @ 292961c)

Two coupled bugs in `confirm_trade` (server-module/src/trading.rs:429-648):

- **Item destruction:** apply loop (619-630) does per-transfer `consume_one(from)` → `grant_item(to)`,
  initiator-items-first (build_swap_plan, rules.rs:215-227). Same-item bilateral swap: counterparty's
  credit (transfer 1) precedes its debit (transfer 2). `grant_item` (inventory.rs:44-47) monotone-clamps —
  at count==9999 the whole grant is dropped. Netted headroom (9999−20+15=9994) passes; apply drops the
  +15 at cap then debits −20 → 9979; 15 destroyed silently.
- **Currency false-reject (latent):** headroom block passes RAW balances (trading.rs:561-562) to
  `check_headroom` while items are netted (511-560, ADR-0113). `validate_proposal` permits currency on
  BOTH sides, so a bilateral currency swap near MAX_BALANCE is falsely rejected today. The un-netted
  check *masks* currency clamp-destruction; netting it (17.5b-2) without debits-first (17.5b-1) would
  *introduce* destruction (`apply_grant` saturates at MAX_BALANCE). **The two fixes are inseparable.**

`check_headroom` (rules.rs:265-314) is already correct given debits-first + netted inputs.

## Design

**Additive game-core seam:** `SwapPlan::ordered_steps() -> Vec<ApplyStep>` emitting ALL debits before
ALL credits; the shell's item+currency apply loops collapse into ONE loop with an exhaustive `match`
dispatching to the fixed `consume_one`/`spend_currency`/`grant_item`/`grant_currency` primitives.

```rust
pub enum ApplyStep {
    ItemDebit      { from_initiator: bool, item_id: u32, qty: u32 },
    CurrencyDebit  { from_initiator: bool, amount: u64 },
    ItemCredit     { to_initiator: bool, item_id: u32, qty: u32 },
    CurrencyCredit { to_initiator: bool, amount: u64 },
}
```

- **Credit variants carry `to_initiator` (the RECEIVER), not `from_initiator`** [B-1/F2/F5]: debit arms
  read "consume from `party(from_initiator)`", credit arms read "grant to `party(to_initiator)`" — NO
  inversion anywhere in the shell match; `ordered_steps()` computes `to_initiator = !from_initiator` at
  emission, and that inversion is executed-tested per-party (A1). Self-documenting on both arms.
- Monster transfers are ownership flips (no cap/clamp) — they stay in their own loop (596-616),
  BEFORE the step loop, untouched. `ApplyStep` covers only items + currency.
- **Currency netting stays shell-side, symmetric with items:** pass
  `wallet_balance(x).saturating_sub(offer.x_currency)` (that party's OUTGOING) to `check_headroom`
  (replacing raw balances at 561-562). Rationale: item netting is already shell-side (m16.5b precedent);
  `check_headroom` is a pure receiver-centric checker that trusts its inputs; moving netting inside
  would change its 8-arg signature and ~20 existing tests for no SSOT gain.
- **Netting is cap-headroom-only, NOT afford-ability** [M-2/F3]: when outgoing > live balance,
  `saturating_sub` yields 0 and the cap check may pass; the broke-sender rejection site remains
  `spend_currency` Err inside the step loop → whole-reducer transaction rollback (SpacetimeDB
  atomicity). This is existing behavior, unchanged by this slice; ADR-0123 documents it explicitly and
  A3 has a boundary test pinning the division of labor.
- **Why the seam is executed-testable:** reducers can't run in unit tests. Hoisting the ordering
  *decision* into game-core means executed tests walk the identical `Vec<ApplyStep>` production
  consumes; the source-guard then only proves the shell iterates that sequence without re-sorting,
  discarding, or shadow-looping [M-1/F4 — needle spec below].

**Rejected:** (a) split debit/credit Vec fields on SwapPlan — breaks every existing `item_transfers`/
`currency_transfers` consumer, four loops can't be observed as one ordering contract; (b) shell-only
reorder + source-scan — spec rejects scan-only for 17.5b-3; no executed test kills a reorder mutant;
(c) fallible `grant_item_exact`/`grant_currency_exact` — touches forbidden files (inventory.rs/
economy.rs) and does not fix the false-reject.

## Tasks (dependency order, tester ≠ implementer)

**Phase A — tester (RED, game-core rules.rs `mod tests`):**
- A1: executed 17.5b-3 tests driving real `ordered_steps()` over an in-memory model keyed
  **per-party**: `ModelState { items: HashMap<(bool /*is_initiator*/, u32), u32>, wallets: … }` [F2].
  Model credit op MUST assert BEFORE clamping [F6]:
  `assert!(old + qty <= MAX_ITEM_STACK, "tripwire: credit would exceed cap — debits-first violated")`
  — the `.min()` may follow, but the assert fires first (a clamp-that-engages IS the regression signal).
  Debits use `checked_sub().expect(…)`. Assertions per case: **per-party exact deltas** (initiator loses
  exactly initiator_items/currency, gains exactly counterparty's, and vice versa — kills identity-swap
  [F2/B-1]) AND aggregate conservation. Cases: same-item bilateral swap, receiver at 9999 and at 9998
  → conserved exactly, no false Err; genuine over-cap net → `check_headroom` Err **before any walk**
  (model untouched trivially — see honesty note).
  **Honesty note [B-2]:** "both inventories unchanged" is proven by check-before-walk; production
  partial-failure safety rests on SpacetimeDB reducer-Err transaction rollback, which is a platform
  guarantee NOT exercised here — stated in the test doc-comment and ADR-0123, not silently implied.
- A2: proptest — offers generated **constructively** so netted `check_headroom` passes (qty bounded by
  remaining headroom, e.g. `qty in 0..=(cap − netted_current)`), NOT `prop_assume!` rejection sampling
  [N-2/F10] — the near-cap region must be exercised, not filtered out. Property: walking
  `ordered_steps()` never trips the model tripwire at ANY intermediate step and per-party + aggregate
  totals are conserved exactly.
- A3: `ordered_steps()` contract teeth:
  - partition: all `ItemDebit`/`CurrencyDebit` strictly before any `ItemCredit`/`CurrencyCredit`;
  - **step-content parity [F1]:** for EVERY ItemTransfer in the plan, exactly ONE ItemDebit with
    (`from_initiator`, `item_id`, `qty`) equal to the transfer AND exactly ONE ItemCredit with
    (`to_initiator == !from_initiator`, same `item_id`, **same `qty`**); identically for currency.
    Kills the qty-divergence value-printing mutant and the emission-side identity swap;
  - zero-currency plan emits NO currency steps (build_swap_plan's >0 filter mirrored — no phantom
    steps) [N-1];
  - currency netting sensitivity (17.5b-2): asymmetric bilateral case (initiator sends 100 /
    counterparty sends 200, distinct balances) constructed so `check_headroom`'s outcome FLIPS if the
    two outgoing subtrahends are swapped [F7];
  - broke-sender boundary [M-2/F3]: netted balance 0 (outgoing > live) with incoming > 0 →
    `check_headroom` PASSES (cap-wise), doc-comment names `spend_currency` as the rejection site.

## Anti-patterns (named)
1. Implementer editing gating tests (tester owns A/B; gate changes = tester task with rationale).
2. Parallel-model drift — executed tests drive REAL `ordered_steps()`; tripwire asserts BEFORE min().
3. Needle over-fitting to formatting — whitespace-normalize; `cargo fmt` must not flip any gate.
4. Interleaving via monster transfers — they are not debits/credits; keep out of ApplyStep.
5. Breaking check_headroom-before-build_swap_plan ordering.
6. Netting asymmetry (items vs currency, or one side only) — sensitivity test + needle pin both.
7. Presence-only needles — every scan needle is loop-consumption/pairing-form with bad fixtures.

## Residual risks (PR flags)
- Shell match-arm dispatch (which identity feeds each primitive) is direct-read under `to_initiator`
  naming and needle-pinned for netting, but a deliberate `party(!step.to_initiator)` inversion is
  beyond scan reach — accepted seam boundary; per-party executed tests pin the emission side.
- Model-mirror fidelity: conservation proven by per-party deltas + tripwire-before-clamp, not clamp value.
- "Inventories unchanged" on reject = check-before-walk + SpacetimeDB rollback (platform guarantee,
  not exercised) — stated honestly in ADR + test docs.
- `ordered_steps()` allocates a Vec per confirm — negligible; iterator variant rejected (YAGNI).
- Nightly mutation caps may need honest re-baseline (ADR-0118 procedure) — run targeted mutants first.
