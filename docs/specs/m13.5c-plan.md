Also via `spacetime sql`: the view executes as the CLI caller → no cross-identity rows (module owner can still read the TABLE directly — ADR wording must scope the closed channel to client subscriptions).

The connection.ts delete handler must apply NET-EFFECT semantics: remove ONLY if the deleted row matches the currently-stored row (compare npcEntityId + currentNodeId); order-independent within a batch. Implement as a small PURE helper (unit-tested; e.g. alongside the dialogue model), called from the connection.ts lambda (which stays coverage-excluded shell). This supersedes the "~10 mechanical lines" client-delta estimate.
5. **Visibility-flip republish over an existing db WITHOUT `--delete-data`: succeeded** ("Updated database") — caveat: the flipped table was EMPTY at flip time; live-row variant untested (record honestly in ADR; nightly republish smoke is the deploy-time net).
