# m17.5c build plan — shop buy/sell: reject-not-destroy at receiver caps

**Slice:** m17.5c (M17.5 §17.5c, EARS 17.5c-1/-2) · **ADR:** 0124 (supervisor-assigned) ·
**Branch:** feat/m17.5c-shop-headroom off master `90090fc` · **Date:** 2026-07-18

## Problem
`buy` (economy.rs:97-150) does `spend_currency` then infallible `grant_item` — at
`MAX_ITEM_STACK` (9999) the grant is dropped entirely (inventory.rs monotone clamp) and the
player has already paid. `sell` (economy.rs:165-230) does `consume_one × qty` then
`grant_currency`, which saturates at `MAX_BALANCE` (999_999_999) — items are consumed for
clamped proceeds. Neither path has a headroom reject. This is the ADR-0113 reject-not-clamp
invariant (established for the trade receiver-cap path, extended to same-item swaps by
ADR-0123) not yet propagated to the shop paths.

## Design (functional core / imperative shell)
- **game-core `trading/rules.rs`:** factor two pure single-receiver primitives out of the
  8-arg `check_headroom` (rules.rs:354-403), which then DELEGATES to them, so the cap
  comparison lives exactly once per axis (SSOT):
  - `check_item_headroom(current_count: u32, incoming_qty: u32, item_id: u32) -> Result<(), TradeError>`
    — `current.saturating_add(qty) > MAX_ITEM_STACK → Err(ItemStackCapExceeded { item_id })`.
  - `check_currency_headroom(balance: u64, incoming: u64) -> Result<(), TradeError>`
    — keeps the trade path's `incoming > 0` gate; `balance.saturating_add(incoming) >
    MAX_BALANCE → Err(CurrencyCapExceeded)`. Behavior-preserving delegation: signature and
    error variants of `check_headroom` unchanged; all ~15 existing positional-arg tests stay
    green untouched.
  - **Gate ownership (plan-review B-1/F11):** the `incoming > 0` early-return is the FIRST
    line of `check_currency_headroom`; `check_headroom` delegates UNCONDITIONALLY with no
    gate of its own and passes the balance through EXACTLY (no `.min(MAX_BALANCE)`
    normalization). A direct primitive test `check_currency_headroom(MAX_BALANCE + 1, 0) →
    Ok` pins the gate independent of delegation (a normalizing delegation is otherwise
    invisible to the existing check_headroom skip-guard tests at rules.rs:1317/1348).
    Primitive doc must state absolute-balance policing is NOT its contract (red-team F1).
- **server-module `economy.rs` (imperative shell):**
  - `buy`: after the trade-escrow currency guard, BEFORE `spend_currency` — read the caller's
    RAW current stack count for `item_id` (same lookup shape `sell` already uses) **with
    `.unwrap_or(0)` for the no-row/new-buyer case (B-2)** and
    `check_item_headroom(current, qty, item_id).map_err(|e| e.to_string())?`.
  - `sell`: after the trade-escrow item guard, BEFORE the `consume_one` loop — read raw
    `wallet_balance(ctx, me)` **at the headroom-check site (not hoisted above the escrow
    guard, W-1)** and `check_currency_headroom(balance, total).map_err(|e| e.to_string())?`.
    (`checked_mul` already precedes this slot, so `total` exists; overflow rejects first —
    defense-in-depth, note in flow comment, F10.)
  - **Raw, not escrow-netted** reads: `grant_item`/`grant_currency` credit the raw
    stack/balance; escrow is a spend-lock, not a receive-lock. (Contrast: the trade path nets
    because it is debit-then-credit — trading.rs:569-578; do not touch.)
  - Boundary semantics: exact-fill to cap is `Ok`; reject only on strict `> MAX` (matches
    `check_headroom`).
- **Export surface (mechanical, touches-delta):** add the two names to
  `trading/mod.rs` `pub use rules::{…}` and the crate-root `pub use` in `lib.rs`
  (`MAX_BALANCE` stays unexported — encapsulated by the primitive).

## Proof-of-teeth
Executed game-core boundary tests (both axes, both directions of each boundary) ·
source-scan order pins with TEETH messages · eval bad/good fixture self-checks that bite
before scanning real source.

## Mutation obligations
`>`→`>=`/`<`/`==` on both cap comparisons killed by reject/accept boundary pairs;
`&&`→`||` on the currency gate killed by zero-incoming accept; delegation covered by existing
check_headroom suite. mutate-server cap stays 299 (ADR-0118) — add tests, never raise the cap
for new code.

## Anti-patterns (named)
Clamp-instead-of-reject · escrow-netting the receive-headroom · cap literals/comparisons in
economy.rs · implementer editing gating tests · `new RegExp()` in evals · touching
evals/run.mjs / CHANGELOG.md / docs/adr/README.md · any ADR number ≠ 0124 · changing
`check_headroom`'s signature/variants.

## Risks
clippy pedantic (`#[must_use]` + doc comments on new fns) · eval teeth fixtures must be
brace-balanced for `extractFunctionBody` · split-literal needle typos = silent false-green
(cross-check against wired source) · headroom slot must be after `checked_mul` (total exists)
and before spend/consume.
