## 0. Problem restated (verified @ this worktree)

The "one ongoing battle per player" invariant is enforced against only the
`player_identity` role across every PvE path, while `pvp::is_in_ongoing_battle`
(pvp.rs:96-110) is the *only* predicate that checks BOTH roles. A PvP battle
indexes side B under `opponent_identity`, so the accepting player passes every
PvE-path guard and can open a second concurrent battle with the same party —
enabling HP-laundering (snapshot HP in a wild battle, lose the PvP, flee to
restore), plus mid-PvP heal/evolve/fuse.

**Confirmed gap sites (all `player_identity`-role-only):**

| Site | File:lines | Callable | Role checked |
|---|---|---|---|
| `start_battle` inline | battle.rs:109-120 | prod (practice) | player only |
| `begin_encounter` inline | battle.rs:331-340 | internal (movement_tick, start_wild_battle) | player only |
| `start_wild_battle` inline | battle.rs:499-510 | `#[cfg(dev_reducers)]` | player only |
| `heal_party` inline | raising.rs:284-293 | prod | player only |
| `evolve` | evolution.rs:66-69 | prod | player only (`player_identity().filter` → `reject_if_in_battle`) |
| `fuse` (×2, owners a+b) | evolution.rs:216-223 | prod | player only |
| `challenge_pvp` / `accept_challenge` | pvp.rs:648/655, 785/790 | prod | **both (correct)** — via `is_in_ongoing_battle` |

**The correct predicate to hoist** (pvp.rs:96-110): player-arm = any `Ongoing`;
opponent-arm = `Ongoing && b.opponent_identity != WILD_IDENTITY`. The
`!= WILD_IDENTITY` refinement MUST be preserved exactly — it stops a wild/practice
battle's `WILD_IDENTITY` sentinel opponent from spuriously matching a real caller
who happens to equal the sentinel, while still catching that battle's real side-A
owner via the player-arm.

## 1. Design

### 1.1 Hoisted helper — location, signature, split (EARS 17.5a-1)

**Decision: two-function split in `guards.rs`** — a thin context wrapper delegating
to a pure iterator-core, mirroring the established `reject_if_in_battle` /
`reject_if_monster_in_trade` shape (pure predicate + call-site-chained iterators).

```rust
// guards.rs — new, adjacent to reject_if_in_battle.

/// Pure core: is `identity` in an Ongoing battle in EITHER role?
/// `as_player` = battle rows indexed by player_identity == identity.
/// `as_opponent` = battle rows indexed by opponent_identity == identity.
/// The opponent arm keeps the WILD_IDENTITY refinement (ADR-0122 D1): a wild/
/// practice battle's sentinel opponent must NOT match; the battle's real side-A
/// owner is still caught by the player arm.
pub(crate) fn is_in_ongoing_battle_either_role(
    as_player: impl Iterator<Item = impl std::borrow::Borrow<crate::schema::Battle>>,
    as_opponent: impl Iterator<Item = impl std::borrow::Borrow<crate::schema::Battle>>,
) -> bool { /* player.any(Ongoing) || opponent.any(Ongoing && opp != WILD_IDENTITY) */ }

/// Thin context wrapper: the single SSOT called by every reducer (PvE + PvP).
/// Replaces the private pvp.rs copy (touches-delta).
pub(crate) fn is_in_ongoing_battle(ctx: &ReducerContext, identity: spacetimedb::Identity) -> bool {
    is_in_ongoing_battle_either_role(
        ctx.db.battle().player_identity().filter(identity),
        ctx.db.battle().opponent_identity().filter(identity),
    )
}
```

**Why this split (alternatives evaluated):**
- **A. Single `fn(ctx, identity) -> bool`** (verbatim move of pvp.rs's fn):
  rejected — not unit-testable without a `ReducerContext` (impossible in this
  crate), so the both-role logic (including the WILD refinement) would be pinned
  ONLY by source-scan + eval, with no executed both-role matrix. EARS 17.5a-3
  wants an executed laundering-path assertion.
- **B. Two-function split (chosen):** the pure core takes two iterators exactly
  like `reject_if_in_battle`, so the full both-role matrix + WILD-sentinel row +
  side-B-two-Ongoing-rows laundering assertion run as unit tests over the core.
  The thin wrapper is a 4-line delegation pinned by source-scan only (acceptable
  — it has no branch logic to mutate). This is the exact pattern the codebase
  already uses; **no new shape invented** (YAGNI honored).
- **C. Reuse `reject_if_in_battle` with chained iterators** (as evolve/fuse will,
  §1.3): rejected for the identity-role check because `reject_if_in_battle` keys
  on `monster_id ∈ party/opponent_monster_ids`, NOT on the caller's *identity
  role* — and it lacks the `!= WILD_IDENTITY` refinement. Different predicate;
  do not conflate. (evolve/fuse legitimately use `reject_if_in_battle` because
  they guard a *monster*, not an identity — see §1.3.)

**SSOT enforcement:** `pvp.rs` deletes its private `is_in_ongoing_battle`
(pvp.rs:96-110) and imports `guards::is_in_ongoing_battle`; the two call sites
(pvp.rs:648/655 challenge_pvp, :785/790 accept_challenge) now reference the shared
fn unchanged in behavior. **Anti-pattern guarded against:** a *partial hoist*
(guards helper added but pvp.rs keeps its copy = two SSOTs) — the eval MUST assert
`is_in_ongoing_battle` is DEFINED exactly once across server-module and that pvp.rs
does NOT contain a local `fn is_in_ongoing_battle`.

### 1.2 PvE inline call-site rewrites (EARS 17.5a-1)

Replace each `player_identity().filter(x).any(|b| ...Ongoing)` inline block with
the shared wrapper. **Error-message strings stay byte-stable** (no behavioral
churn, keeps existing log-reject reason vocabulary):

- **`start_battle`** (battle.rs:110-116) → `if is_in_ongoing_battle(ctx, me) {`
  keep `"already in an ongoing battle"` + `log_reject("start_battle", me, &e)`.
- **`begin_encounter`** (battle.rs:332-340) → `if is_in_ongoing_battle(ctx, player_identity) {`
  keep `"already in an ongoing battle"` (no log_reject today — leave as-is).
- **`heal_party`** (raising.rs:285-293) → `if is_in_ongoing_battle(ctx, me) {`
  keep `"cannot heal during an ongoing battle"` (distinct string — preserve).
- **`start_wild_battle`** (battle.rs:500-510) → `if is_in_ongoing_battle(ctx, me) {`
  keep `"already in an ongoing battle"`. **INCLUDED** (decision, §4 D3): it is
  `#[cfg(dev_reducers)]` = not a prod vector, but it lives in the same file, is a
  trivial one-block swap, and leaving one duplicated role-check contradicts the
  SSOT/"zero duplicated role logic left anywhere" invariant. Cheap uniformity win.

### 1.3 evolve / fuse chain fix (EARS 17.5a-2)

These guard a *monster* (via `reject_if_in_battle`, which scans `monster_id`),
not an identity — so the fix is to CHAIN the `opponent_identity` iterator into the
existing call, exactly mirroring the m16.5a trade-interlock fix (trading.rs already
chains initiator/counterparty for the trade guard in this same code path):

- **`evolve`** (evolution.rs:66-69):
  ```rust
  reject_if_in_battle(
      ctx.db.battle().player_identity().filter(m.owner_identity)
          .chain(ctx.db.battle().opponent_identity().filter(m.owner_identity)),
      monster_id,
  )?;
  ```
- **`fuse`** (evolution.rs:216-223): same chain for BOTH calls (owner a, owner b).

Note: `reject_if_in_battle` here needs no `!= WILD_IDENTITY` refinement — it keys
on `monster_id ∈ opponent_monster_ids`, and a wild battle's side B has
`opponent_monster_ids == []` (battle.rs:442), so an owned monster can never appear
in a wild opponent slot. The chain simply catches the case where the owner's
monster is side B of a *PvP* battle. (VERIFIED at plan review: side-B PvP monsters
ARE listed in `opponent_monster_ids` — pvp.rs:245 `opponent_monster_ids:
opponent_party` — so the chain is a real guard, not vacuous.)

**Seam parity (plan-review B-1, BINDING):** the test seams `evolve_seam`
(evolution_tests.rs:1419) and `fuse_seam` (evolution_tests.rs:1508/1512) reproduce
the player-identity-only filter verbatim. The TESTER (T1) MUST update both seams to
the both-role chain AND add side-B seam tests (owner appears ONLY as
`opponent_identity` of an Ongoing PvP row → evolve/fuse rejected through the seam).
These seam tests go GREEN immediately once the seam is updated (they execute
`reject_if_in_battle` directly — this is expected, they are the *executed*
regression spec); the RED gate that forces the PRODUCTION change is the C2 eval
criterion. Without the seam update, the side-B unit coverage for evolve/fuse is
illusory.

### 1.4 movement.rs — untouched, with recorded residuals (EARS 17.5a-4)

`movement_tick` reaches `begin_encounter` at movement.rs:276 and swallows its Err
(`let _ =`). Post-fix, `begin_encounter` rejects a side-B caller → **no second
battle row is ever created**. The player-only pre-check at movement.rs:251-259
(skips the encounter roll for player-role Ongoing) is NOT a security boundary and
is intentionally left untouched (EARS 17.5a-4: inheritance through
`begin_encounter` suffices; movement.rs is outside the declared touches set).

**Two behavioral residuals to record in ADR-0122 (NOT fixed here):**
1. **Wasted RNG draw:** a side-B player on grass mid-PvP is missed by the pre-check
   (player-role only), so `ctx.random()` (movement.rs:272) draws ONE encounter seed
   before `begin_encounter` rejects. Harmless (determinism unaffected — the seed is
   consumed identically whether or not the encounter is created; the tick swallows
   the Err), but note it so a future reader doesn't file it as a bug.
2. **Warp-guard same-class inconsistency:** the warp battle-guard (movement.rs:208-222)
   is ALSO player-role-only, so a side-B player CAN warp mid-PvP. This is a
   *separate* invariant (warp-during-battle, not battle-creation) and out of scope;
   record as an ADR residual / candidate follow-up, do not fix in this slice.

### 1.5 Classification guard (EARS 17.5a-5) — replaces the deferred BattleKind column

Per spec Decision A (amended): NO `BattleKind` column. Instead a new eval criterion
pins the `is_ranked_pvp` (guards.rs:284-286) negative-space classifier against the
**three battle-creation insert sites**, so a future battle source can't silently
rate as ranked:

- battle.rs:264 (`start_battle`) → `opponent_identity: opponent_identity` where the
  provenance gate (battle.rs:82) has already forced `opponent_identity ∈ {me, WILD_IDENTITY}`.
- battle.rs:436 (`begin_encounter`) → `opponent_identity: WILD_IDENTITY`.
- pvp.rs:239 (`accept_challenge`) → `opponent_identity` = a real player.

The criterion asserts: every `ctx.db.battle().insert(Battle {` site's
`opponent_identity` provenance is one of {`WILD_IDENTITY` sentinel, the caller
(self/practice, gated), a real accept-challenge player}, AND `is_ranked_pvp`'s
body is the two-clause sentinel form (`!= opponent_identity && != WILD_IDENTITY`),
so adding a new insert site with a novel `opponent_identity` provenance forces the
author to touch both `is_ranked_pvp` and this eval. Bad-fixture proof-of-teeth
required (§2.3).

### 2.1 Unit tests — `guards_tests.rs` (pure core, START RED)

Over `is_in_ongoing_battle_either_role`, reusing/extending `make_test_battle`
(guards_tests.rs:234 — note it currently hardcodes `opponent_identity = [0;32]`;

exist yet):

| Test | Asserts | Kills mutant |
|---|---|---|

| `either_role_player_ongoing_true` | player-arm has Ongoing → `true` | dropped player arm |
| `either_role_opponent_ongoing_true` | **EMPTY player arm** + opponent-arm has Ongoing with **non-WILD** opponent (e.g. player `[1;32]`, opponent `[2;32]`) → `true` | **dropped opponent arm** (the core bite) |
| `either_role_opponent_wild_sentinel_false` | **EMPTY player arm** + opponent-arm row has `opponent_identity == WILD_IDENTITY` → `false` | dropped `!= WILD_IDENTITY` refinement (would flip `true`) |

| `laundering_two_ongoing_rows` | caller is side-A of an Ongoing wild battle (player arm) AND side-B (opponent, non-WILD context) of a distinct Ongoing PvP battle (opponent arm) → `true`; ALSO assert the PvP-row-only variant with empty player arm → `true` (the exploit precondition executed) | the whole gap |

The **`either_role_opponent_wild_sentinel_false`** test is the one that pins the
refinement EARS 17.5a-1 mandates keeping.

**Fixture trap (plan-review N-1 / red-team F6, BINDING):** `make_test_battle`'s
hardcoded `opponent_identity = [0u8;32]` **IS `WILD_IDENTITY`** (lib.rs defines the
sentinel as `Identity::from_byte_array([0u8;32])`). Reusing it unmodified for the
opponent-arm tests makes them vacuous or wrong-reason-green. T1 MUST parameterize
the opponent identity (or mirror the `make_pvp_test_battle` pattern at
guards_tests.rs:401) and MUST use an **empty player arm** for
`either_role_opponent_ongoing_true` / `_wild_sentinel_false` so the opponent arm is
the only possible source of the result — a broken opponent-arm cannot be masked by
the player arm.

## 4. ADR-0122 outline

Header (canonical ADR-0104 block; supervisor-assigned 0122):
```
# 0122 — m17.5a: unify the ongoing-battle guard to both roles (SSOT hoist)
**Status:** Accepted
**Date:** 2026-07-17
**Slice:** m17.5a (M17.5 tenth-review residuals §2 17.5a)
**Supersedes:** —
**Amends:** ADR-0119 (subsumes/widens the never-scheduled m17-fix-sideb-guards residual)
**Subsystems:** battle, security-authz
**Decision:** hoist pvp::is_in_ongoing_battle into guards.rs as an SSOT (pure
iterator-core + thin ctx wrapper); all PvE + PvP callers share one both-role
definition; evolve/fuse chain the opponent_identity iterator; classification
pinned by eval, not a BattleKind column.
```

**Decisions to record:**
- **D1 — Hoist location + pure-core split.** Why guards.rs, why two functions
  (unit-testability of the both-role matrix + WILD refinement), why not verbatim
  move. Keep `!= WILD_IDENTITY` refinement verbatim.
- **D2 — movement_tick inheritance.** begin_encounter rejects side-B → no second
  row; document the **wasted-RNG-draw** (one `ctx.random()` before rejection,
  determinism-neutral) and why the pre-check is left as a non-security fast-path.
- **D3 — start_wild_battle inclusion.** dev-only, but folded in for SSOT
  uniformity (zero duplicated role logic remains).
- **D4 — warp-guard residual.** movement.rs:208-222 warp guard is player-role-only;
  side-B can warp mid-PvP — separate invariant (warp-during-battle ≠
  battle-creation), recorded as a candidate follow-up, out of scope here.
- **D5 — eval-instead-of-BattleKind.** Reference spec Decision A (amended): the
  17.5a-5 classification eval provides the silent-rating safety a `BattleKind`
  column would, at a fraction of the cost; the additive column is deferred to the
  milestone that first introduces a third battle kind (raids/friendlies, post-gate,
  may be cut). YAGNI. Note the per-site (file, enclosing-fn, RHS-form) allowlist
  design and why bare token allowlists were rejected (alias bypass, red-team F5).
- **D6 — touches-delta pvp.rs.** Record that the hoist required editing pvp.rs
  (delete local fn, import shared), outside the declared touch set.
- **D7 — care/train mid-battle residual (red-team F1/F3, ACCEPTED, not fixed here).**
  `care` and `train` (raising.rs) have NO battle guard in either role: mid-PvP they
  mutate `bond`/`evolves_to`/EVs/derived stats of a live PvP participant. NOT an
