## 0. Grounding facts (verified in SDK source + codebase ÔÇö cite, do not re-derive)

- **SDK reducer promise:** `conn.reducers.X(args): Promise<void>` ÔÇö resolves on Ok/OkEmpty, rejects `SenderError(errorString)` on reducer `Err`, `InternalError` otherwise (`node_modules/spacetimedb/src/sdk/db_connection_impl.ts:1124-1141`, twin at 1165-1182). **No per-reducer `onX` callbacks exist in SDK 2.6** (main.ts:707-709 already documents this; m13d's buy/sell use try/catch on the promise). `SenderError`/`InternalError` are root-exported (`src/index.ts:2` Ôćĺ `lib/errors.ts`).
- **No settle on drop:** `ws.onclose` only sets `isActive=false` + emits `disconnect` (`db_connection_impl.ts:331-334`); `#reducerCallbacks` are settled ONLY on message receipt (`:999-1000`). **In-flight reducer promises never settle after a drop** Ôćĺ the input freeze MUST be event-driven (disconnect), never promise-driven; the `.catch` fires only on a real server `Err`.
- **Sends while disconnected are silently queued** on the dead instance (`#sendEncodedMessage` else-branch) and never flushed (no auto-reconnect on the raw path) ÔÇö hence "no local prediction against a dead connection".
- **No auto-reconnect on the raw builder path.** The SDK's `ConnectionManager` (React refcount layer, `connection_manager.ts:49-60`) does exponential backoff `min(1000┬Ě2^attempt, 30_000)` but is NOT root-exported Ôćĺ app-level rebuild required (ADR-0085).
- **The bug:** `predictor.ts:191` prunes `#pending` only by `p.seq > ackedSeq`; a rejected `enqueue_move` (seq N, never acked; server rolls back its seq bump on Err) survives forever and replays onto the auth queue at every reconcile Ôćĺ persistent 1-tile offset with `diverged=false`.
- **Forced-reconcile necessity:** when the rejected send is the burst tail, NO further authoritative batch arrives (server state unchanged) ÔÇö the fix must actively re-reconcile from current store state (extract the `store.onBatchApplied` body, main.ts:157-224, into a callable `reconcileFromStore()`).
- **Coverage (hard):** `main.ts` + `connection.ts` are the only sanctioned coverage-excluded shells (exact-set-guarded eval; `vite.config.ts` OUT of touches); client line threshold 96 Ôćĺ all new decision logic in new pure modules with colocated tests.

**Correctness contracts (reviewer):**
- C1 (B-1 + m-5). `reconcileFromStore()` is TOTAL (internal try/catch ÔÇö the single totality source for both callers; batch listener keeps its outer catch as belt). Early-exit when own/player rows are absent (store reset mid-gap) is SAFE but leaves `#pending` dropped while `#queue` still reflects the phantom ÔÇö a transient that self-heals on the next real batch reconcile; documented in ADR-0085 + a code comment at the early-exit.
- C2 (M-1). The `MicrotaskBatcher` lives in the OUTER `connect()` scope ÔÇö ONE batcher reused across rebuilds (a per-build batcher could fire a stale flush after `store.reset()`). `wireTables(conn)` re-registers row handlers per build; `hadSession`, `identity`, policy state, and the batcher live in the outer scope.
- C3 (atomicity justification, for ADR-0085 verbatim): calling reconcile from a `.catch` microtask does NOT violate the reconcile-atomicity rule: a rejected reducer produces NO row events (nothing half-applied is in flight), and any prior transaction's row burst was already coalesced+flushed by the batcher before this turn's microtasks run ÔÇö the store is a complete per-transaction snapshot at `.catch` time.
- C4 (answer d). Freezing only the SEND side is sufficient: during the reconnect gap the old predictor may drain, but `store.reset()` cleared rows Ôćĺ `ownEntityId === undefined` Ôćĺ RenderResolver produces no own entity (blank world is expected + correct during the gap); `resetPredictionState()` on reconnect rebuilds prediction. ADR text.
- C5 (n-2). `reconnectDelayMs` uses `2 ** attempt` (NOT bitshift ÔÇö `1 << 31` breaks); `min(Infinity, MAX)` is safe; test `reconnectDelayMs(1024) === 30_000`.
- C6 (m-2). `dismissDialogue` rejection also RESETS `dismissPending` (else the Escape dismiss is a dead button forever after one rejection) ÔÇö call-site-specific catch.
- C7 (task E, testability). The payload extraction is a pure fn in statusModel: `subscriptionErrorMessage(ctx: unknown): string` (returns `.event.message` when present, else the fallback) ÔÇö unit-tested with/without `.event`; connection.ts calls it.
- C8 (n-3). The status div is created (and `reportError` bound) BEFORE `conn = connect(...)` is assigned in `main()`.
- C9 (m-1/m-6, ADR notes). ADR-0085 records: `err.name` equality (not `instanceof` ÔÇö cross-realm/bundling safety); the spec's "reducer-status callback" language maps to the SDK's promise-rejection mechanism (line-cited); `conn.conn` getter naming kept for call-site compatibility (callers must not cache `conn.conn` across await points ÔÇö comment on the getter).
