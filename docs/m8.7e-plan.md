## 1. EARS (source of truth)
> WHEN a player's battle resolves (SideAWins/SideBWins/Fled/recruit-success) THE SYSTEM SHALL render the terminal outcome frame at least once (the existing `#renderOutcome` path) rather than hiding the overlay the instant `outcome != Ongoing`, and SHALL provide an explicit dismiss (Escape and/or a brief timeout). The Ongoing-only auto-show for active battles is preserved.

## 2. Confirmed facts
- Resolved battles persist: the server `update`s the `battle` row in-place (lib.rs:1640/1700/1729/2063/2095), never deletes it → the resolved row stays in the client `AuthoritativeStore`. No new subscription needed.
- `buildBattleViewModel(battle, …, baitItems=[])` already returns a full VM for a non-Ongoing outcome; `BattleView.#renderOutcome` already maps SideAWins/SideBWins/Fled→"Victory!/Defeat…/Got away safely!". The gap is purely UPSTREAM (which battle is fed to the overlay + dismiss lifecycle).
- `battleId` is a monotonic server auto-inc `bigint` → "max battleId for the player" = the current/most-recent battle.

## 3. Design (functional core / imperative shell)

### 3a. `store.ts` — add `latestPlayerBattle(identity)` (do NOT touch `ongoingBattle`)
Highest-`battleId` row among `playerIdentity === identity`, ANY outcome; `undefined` when none.
**MUST compare with bigint `>` (never `Number()`/`Math.max`)** — ids exceed 2^53 (red-team F9). `ongoingBattle()` stays byte-identical (the `__game` snapshot + M7c tests depend on it).

### 3b. `battleModel.ts` — add a PURE reducer `decideBattleOverlay`
```ts
export type BattleOverlayAction =
  | { readonly kind: 'show'; readonly battle: StoreBattle }  // ongoing auto-shows; terminal shows the outcome frame
  | { readonly kind: 'hide' };
export interface OverlayState { readonly dismissedBattleId: bigint | null; readonly synced: boolean; }
export interface OverlayResult { readonly action: BattleOverlayAction; readonly dismissedBattleId: bigint | null; readonly synced: boolean; }
export function decideBattleOverlay(latest: StoreBattle | undefined, state: OverlayState): OverlayResult;
```
Rules (a pure `(state, input) → (action, nextState)` reducer; total over `{undefined, 'Ongoing', terminal}`):
1. `latest === undefined` → `hide`, state unchanged.
2. `!synced` (first battle observed this session): set `synced = true`. If `latest.outcome !== 'Ongoing'` → **pre-dismiss** (`dismissedBattleId = latest.battleId`) + `hide` (stale-on-login: a row already terminal at first sight was never watched go Ongoing → don't pop). Else (`Ongoing`) → `show`.
3. `synced` (steady state): if `dismissedBattleId === latest.battleId` → `hide` (no re-pop after dismiss / after pre-dismiss). Else → `show` (Ongoing auto-shows; a mid-session terminal — even one coalesced Ongoing→terminal in a single batch — shows its outcome; **defeats red-team F1 under-showing** because there is no seen-Ongoing gate).

### 3c. `main.ts` — thin shell
- Module state (replaces nothing; adds): `let dismissedBattleId: bigint | null = null; let battleSynced = false;`
- `refreshBattle()` (replace body): `const latest = store.latestPlayerBattle(identity); const r = decideBattleOverlay(latest, { dismissedBattleId, synced: battleSynced }); dismissedBattleId = r.dismissedBattleId; battleSynced = r.synced;` then: `show` → `if (boxView?.visible) boxView.hide();` build VM, `battleView.refresh(vm)` (vm null → corrupt-team warn + refresh(null) hides, as today); `hide` → `if (battleView.visible) battleView.hide()`. Stays on `store.onBatchApplied` (no per-rAF work).
- **Escape** (main.ts:132 branch): when `battleView?.visible`, read `latest = store.latestPlayerBattle(identity)`; if `latest && latest.outcome !== 'Ongoing'` set `dismissedBattleId = latest.battleId` (permanent dismiss of the outcome frame); always `battleView.hide()`. → terminal = permanent dismiss; ongoing = existing hide-then-reauto-show (B1/F5). Escape priority (battle > box > movement) unchanged.
- **`onReconnect`** (main.ts:286): add `dismissedBattleId = null; battleSynced = false;` (post-reset, a re-subscribed historical resolved battle is pre-dismissed at first sight again).
- `__game()` snapshot UNCHANGED (golden.spec strips it; recruit.spec is fixme).

### 3d. `battleView.ts` — NO change. `#renderOutcome` + `refresh(vm)` already render the banner.

## 4. Dropped vs the first draft (adjudicated by red-team + simplify)
- **No 5000ms auto-dismiss timer** — Escape-only is spec-compliant ("and/or"); the timer was the source of red-team F2 (stale timer hides a new overlay), F3 (reconnect leak), F6 (re-arm-every-batch never fires). YAGNI; trivially re-addable as a keyed `setTimeout` if playtest wants it.
- **No `seenOngoing` Set** — replaced by `synced` boolean + first-sight pre-dismiss → kills F1 under-showing AND F8 unbounded-Set.

## 5. Tasks (tester writes RED; specialist implements; verifier checks not-weakened)
- **T1 `latestPlayerBattle`** (store.test.ts): max-battleId for identity (any outcome incl. terminal); `undefined` when none; filters by `playerIdentity`; **bigint comparison** (a `Number()`-coercing impl picks the wrong winner above 2^53 — bite it). Existing `ongoingBattle` tests still green.
- **T2 live resolve shows outcome** (battleModel.test.ts): synced terminal, not dismissed → `show` with that battle.
- **T3 stale-on-login** : `!synced` + terminal → `hide` + `dismissedBattleId = id` + `synced = true`. Next call (now synced, dismissed===id) → `hide`.
- **T4 no re-pop after dismiss**: synced + terminal + `dismissedBattleId === id` → `hide` (not show).
- **T5 ongoing auto-show preserved**: `!synced`+Ongoing → `show` + `synced=true`; synced+Ongoing (id≠dismissed) → `show`. `undefined` → `hide`.
- **T6 coalesced/mid-session resolve (F1 teeth)**: synced + terminal + `dismissedBattleId` is a DIFFERENT (older) id → `show` (the new terminal still shows even though never individually seen Ongoing).
- **T7 total/exhaustive**: every `(outcome ∈ {undefined,Ongoing,SideAWins,SideBWins,Fled,unknown}) × (synced?, dismissed?)` returns a defined action, never throws.
- **T8 recruit.spec.ts**: update the `test.fixme` note ONLY to record the M9c bait deferral (no behavior, stays fixme).

main.ts shell wiring (Escape, onReconnect, refreshBattle switch) is e2e-territory; its decision logic is fully proven by T2–T7 at the pure layer.
