## Decisions (adjudicated from planner + reviewer + red-team + simplify lenses)

1. **Held-key model = frame-loop re-issue (preserve continuous movement).** The literal `e.repeat` guard alone would regress continuous held movement to tap-per-tile (a player-facing change; ¬ß6 says smoothness is the only intended change). The e2e drives `__game().step()` directly so it would NOT catch the regression. Chosen model (the intended ADR-0013 realization, using the predictor's `lastQueuedDir`/`queueDepth` accessors "for exactly this dedup"):
   - keydown(movement, **non-repeat**) ‚Üí `step(dir)` **immediately** (direct enqueue, no dedup ‚Äî preserves first-step latency AND deliberate double-tap) + register the held dir.
   - keydown with `event.repeat === true` ‚Üí **ignored at the TOP of the handler** (also fixes the KeyB open/close flicker on held B, red-team F9).
   - frame loop, before `drain`: re-issue the active held dir **deduped** against `predictor.lastQueuedDir` (`active !== undefined && active !== lastQueuedDir ‚Üí step(active)`), bounded by enqueue's cap/backpressure. Suppressed while an overlay is visible.
   - keyup(movement) ‚Üí release the dir; window blur ‚Üí clear; `onReconnect` ‚Üí clear.
2. **`#pending` bound = BACKPRESSURE, not drop-oldest.** Drop-oldest is **desync-unsafe**: dropping an in-flight `Clear`/`SetMove` corrupts reconcile replay ‚Üí RTT-window rubberband (ADR-0013 violation; reviewer B-1 + red-team F2/F11). Instead: `enqueue` declines (returns `undefined`, no push/`#record`/seq) when `#pending.length >= MAX_PENDING` ‚Äî same shape as the existing `#queue` cap. **No ops dropped ‚Üí reconcile replay uncorrupted ‚Üí desync-safe**; directly enforces ADR-0013.5. Does NOT touch `setMove`/`clearQueue`/`reconcile` ‚Üí existing M3b tests unaffected. `MAX_PENDING` = a named constant, a generous degenerate-no-ack backstop normal ack cadence never reaches (reconcile keeps `#pending ‚âà 0`). Under sustained no-ack, prediction freezes at the bound (correct ‚Äî don't predict into the void) and self-heals when an ack arrives.
3. **CUT the fresh-press latch** (simplify + red-team): redundant once keydown enqueues directly ‚Äî a deliberate double-tap re-enqueues via the direct keydown path; dedup governs only the held continuation.
4. **Held state = a pressed-dir STACK, not a scalar** (red-team F3): two keys held, release one ‚Üí fall back to the still-held dir (no stuck). Suppress-re-issue-while-overlay (don't clear) so movement resumes after an overlay closes with the key still held (red-team F5).
5. **Robustness:** `speciesMap()`/`skillMap()` ‚Üí `new Map(this.#species)` / `new Map(this.#skills)` (callers `refreshBox`/`refreshBattle` consume inline, don't retain ‚Üí safe). `buildBattleViewModel` ‚Üí add `|| sideX.active < 0` to both guards (fail soft `null`).

## Anti-patterns to avoid
e.repeat guard with no continuation (one-tile regression, e2e-blind) ¬∑ dropping `Clear`/`SetMove` from `#pending` (desync) ¬∑ deferring the first step to the frame loop (latency/tap-drop) ¬∑ deduping a deliberate fresh re-press ¬∑ scalar heldDir (two-key stuck) ¬∑ `Object.freeze` on a Map (freezes wrapper not entries; freezes the live store map) ¬∑ decision logic in `main.ts` (e2e-only/untestable) ¬∑ editing `client/src/inputGuards.ts` (OUT of touch-set ‚Äî new pure module goes under `client/src/prediction/`).
